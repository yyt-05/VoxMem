# 题目1思考说明

## 题目与选题

### 题目

设计并实现一个面向真实用户痛点的 AI 应用，说明目标用户、用户痛点、功能优先级、技术选型、效果评估方式以及降低第三方 API 调用成本的思路。

### 我的选题：智能语音输入法 VoxMem

我的选题是“语音输入法”，但不是传统意义上只负责把声音转成文字的语音输入工具。我希望做的是一个更接近真实输入场景的智能语音输入工作台：它不仅要能实时转写，还要能处理用户口语中的冗余表达，记住用户反复修正过的个性化名词，并在多人说话环境下尽量减少外来声音干扰。

我把 VoxMem 的核心定位概括为：

```text
VoxMem = 实时语音识别 + 微整理模式 + 本地记忆 + 声纹过滤
```

项目的目标不是做大型会议转写系统，而是解决用户在日常语音输入时最容易遇到的三个具体问题：

1. 用户点击语音输入后准备不足，口语里会出现大量“嗯、那个、不对”等冗余内容。
2. 通用 ASR 难以稳定识别高度个性化的人名、项目名、地名和行业术语。
3. 多人环境下，周围人的声音会被 ASR 误识别进文本，造成语音输入串扰。

## 一、目标用户

### 1. 高频语音输入用户

这类用户经常用语音输入记录备忘、消息、待办、日报、想法或 Prompt。他们希望语音输入比打字更快，但现实中往往还要花时间修改语气词、重复内容和识别错误。

典型用户包括：

- 经常用手机或电脑语音输入的办公用户。
- 需要边走边记录想法的学生、产品经理、销售和创作者。
- 希望用语音快速生成一段可复制文本的人。

### 2. 个性化词汇密集用户

这类用户的输入内容里有大量普通 ASR 不熟悉的词，例如：

- 人名：张立、李珂、王祎。
- 项目名：VoxMem、声纹过滤方案、熔断机制。
- 地名和机构名：某个小区、学校、公司、实验室。
- 行业术语：投研术语、医疗术语、工程术语、法律术语。

这些词往往不是通用语料里高频出现的词。如果只依赖 ASR 的概率命中，第一次识别错后，下一次仍然可能继续错。

### 3. 嘈杂或多人环境中的语音输入用户

用户在办公室、会议室、宿舍、开放空间中输入时，旁边可能有人聊天。传统语音输入通常只要听到声音就识别，不一定能区分“用户本人说的话”和“旁边人说的话”，导致最终文本混入外来内容。

## 二、我想解决的三大用户痛点

## 痛点一：语音输入准备不足，口语冗余影响文本可读性

### 1. 痛点说明

很多用户点击语音输入时，并不会像写文章一样提前组织好语言。他们往往是一边想一边说，因此表达过程中会出现大量语气助词、停顿词、重复词和自我纠正，例如：

```text
嗯那个我们今天先看一下这个接口，不对，先看登录接口，然后那个下午再补测试。
```

传统 ASR 通常会把这些内容完整转写出来。这样虽然“识别了声音”，但文本并不能直接使用。用户还需要手动删除“嗯”“那个”“不对”等内容，整理句子顺序，才能得到真正想要的文本。

这类问题本质上不是 ASR 准确率问题，而是“口语表达”和“可用文本”之间存在距离。

### 2. 我的设计：微整理模式

为了解决这个问题，我设计了“微整理模式”。它的目标不是大幅改写用户内容，而是在保留原意的前提下，自动处理明显的口语冗余。

微整理模式主要处理：

- 删除语气助词：例如“嗯”“啊”“那个”“就是”。
- 删除无效停顿：例如重复出现的“那个那个”“嗯嗯”。
- 处理明确自我纠正：例如“不对，是星期二”“不是 A，是 B”。
- 合并重复表达：例如“确认一下，确认一下接口”整理为“确认接口”。
- 提升基础可读性：补充合理标点，让文本更像可直接使用的句子。

示例：

```text
输入：嗯那个今天是星期一，不对，今天是星期二，下午三点那个同步一下需求。
输出：今天是星期二，下午三点同步需求。
```

### 3. 我完成的实现

项目中已实现三种输出模式：

| 模式 | 作用 |
| --- | --- |
| 原声模式 raw | 保留 ASR 文本，只应用本地记忆替换，不调用 LLM |
| 微整理模式 polish | 清理语气词、重复、明确自我纠正，输出更精炼文本 |
| Markdown 模式 markdown | 将口述的步骤、要点整理成 Markdown 结构 |

微整理模式由后端统一调用 OpenAI 兼容 LLM 接口完成，前端提供模式切换。为了避免 AI 过度发挥，提示词要求保守处理：只删除明显冗余，不添加用户没有说过的新信息。

## 痛点二：个性化名词识别不稳定，需要本地记忆

### 1. 痛点说明

目前 ASR 对常见词的识别已经比较成熟，但用户真实输入中会出现大量高度个性化的词，例如特定的人名、项目名、地名、客户名、部门名和缩写。这些词对某个用户很重要，但对通用模型来说不一定高频。

例如用户经常说：

```text
今天找张立确认 VoxMem 的声纹过滤方案。
```

ASR 可能输出：

```text
今天找张力确认 VoxMem 的声纹过滤方案。
```

“张立”和“张力”读音接近。如果只依赖通用 ASR 的概率命中，系统并不知道这个用户真正想表达的是“张立”。用户每次都需要手动修改，体验会很差。

### 2. 我的设计：本地记忆

我设计了本地记忆机制，让系统从用户的编辑行为中学习个性化词汇。

核心思路是：

1. 用户先得到 ASR 原始转写文本。
2. 用户在最终输入框中手动修正错误。
3. 系统将修正后的内容与原始转写字符做比对。
4. 通过比对定位差异部分。
5. 从差异部分提取有意义的实词，过滤掉虚词和无效词。
6. 将修正关系整理为本地热词或本地替换规则。
7. 把规则存入 SQLite 数据库。
8. 下次遇到相同错误时，系统直接调用本地记忆进行替换。

这里我没有只依赖云端热词概率，而是先落地一套可控的本地替换机制。它更适合 MVP 演示，因为用户修正后可以很快看到效果。

### 3. 本地记忆的技术流程

```text
ASR 原始文本
  -> 用户编辑最终文本
  -> 前端提交 original_text / enhanced_text / corrected_text
  -> 后端计算差异
  -> 提取有效词
  -> 生成 from_text -> to_text 映射
  -> 存入 SQLite hotword_mappings
  -> 下次 ASR 最终文本先应用本地映射
```

示例：

```text
原始转写：今天找张力确认声纹过滤方案
用户修正：今天找张立确认声纹过滤方案
系统学习：张力 -> 张立
下次命中：张力 自动替换为 张立
```

### 4. 我完成的实现

本项目已完成：

- SQLite 本地数据库存储。
- 用户级本地替换映射。
- 纠错记录保存。
- 热词管理展示。
- 替换命中次数统计。
- 最近修正记录展示。
- 输入质量观察面板。

在前端，用户可以看到本地记忆命中的结果，也可以打开热词管理删除错误规则，避免误替换长期存在。

### 5. 为什么用 SQLite

我选择 SQLite 是因为 MVP 阶段不需要复杂的独立数据库服务。SQLite 适合本地演示和快速部署，能保存用户、纠错记录、热词映射和质量指标。后续如果要做账号体系和多端同步，可以把 SQLite 替换为 PostgreSQL 或 MySQL。

## 痛点三：多人语音串扰导致外来内容混入

### 1. 痛点说明

用户进行语音录入时，周边如果存在他人交谈，ASR 可能会把旁人的声音也识别进来，造成转写文本杂乱。

例如用户想输入：

```text
明天下午三点同步需求，重点确认热词命中。
```

但旁边有人说了一句：

```text
帮我拿一下杯子。
```

最终 ASR 可能混成：

```text
明天下午三点同步需求，帮我拿一下杯子，重点确认热词命中。
```

这种错误对语音输入法影响很大，因为用户不是在做会议记录，不需要保留所有人的声音，只希望保留自己的输入内容。

### 2. 我的设计：声纹过滤机制

为了解决多人语音串扰，我在项目中引入了声纹过滤思路：系统尽量锁定目标用户的声音特征，过滤掉旁人的干扰语音，保障转写内容单一准确。

在当前 MVP 中，我采用的是“完整音频文件转写 + 说话人分离”的实现路径：

1. 用户开启“去除周围人声”。
2. 前端录制完整 WAV 音频。
3. 后端将 WAV 上传到七牛 Kodo。
4. 后端把可访问的音频 URL 提交给 DashScope 文件转写接口。
5. 文件转写接口返回按说话人切分的句子。
6. 前端默认保留第一个检测到的说话人的内容。
7. 用户也可以恢复显示全部文本。

### 3. 技术流程

```text
浏览器录音
  -> 前端缓存完整音频
  -> 编码为 WAV
  -> POST /api/transcribe/file
  -> 后端上传到七牛 Kodo
  -> DashScope 文件转写 + 说话人分离
  -> 返回 sentences / speaker_id / full_text
  -> 前端按目标说话人过滤文本
```

### 4. 我完成的实现

项目中已完成可选声纹过滤链路：

- 前端提供“去除周围人声”开关。
- 前端可以录制完整 WAV。
- 后端提供 `POST /api/transcribe/file` 接口。
- 后端接入七牛 Kodo 上传音频。
- 后端接入 DashScope 文件转写和说话人分离。
- 前端支持保留目标说话人文本和恢复全部文本。

需要说明的是，当前版本的声纹过滤不是实时声纹识别，而是文件转写后的说话人分离过滤。它属于 MVP 阶段可演示的多人声音处理方案，后续可以升级为更实时的目标说话人确认或声纹注册机制。

## 三、功能优先级

### P0：本次必须完成

1. **实时语音输入**

   浏览器采集麦克风音频，通过 WebSocket 发送给 Go 后端，后端代理到 DashScope Paraformer 实时 ASR。

   处理“嗯、那个、不对、重复表达”等口语冗余，让最终文本更精炼。

3. **本地记忆**

   保存用户修正记录，生成本地替换规则，存入 SQLite，下次识别时自动应用。

4. **基础工作台 UI**

   页面展示录音按钮、实时识别、最终输入框、输出模式、热词记忆和错误状态。

5. **Mock ASR 模式**

   在没有真实 API Key 的情况下，也能稳定演示主流程。

### P1：增强评审展示

1. **输入质量观察**

   展示修改率、热词命中次数、热词数量、最近修正记录。

2. **热词管理**

   用户可以查看和删除本地替换规则。

3. **音频调试**

   保存 PCM 和 WAV，便于排查麦克风采集和 ASR 问题。

4. **Markdown 模式**

   支持把口述步骤整理成 Markdown 列表。

### P2：可选增强

1. **声纹过滤**

   通过文件转写和说话人分离减少多人声音串扰。

2. **格式偏好记忆**

   记录用户对标点、列表风格等格式偏好。

3. **云端热词同步**

   后续可把本地高频热词同步到 ASR 热词表，让识别阶段也能受益。

### P3：后续方向

1. 系统级输入法集成。
2. 账号体系和多端同步。

4. 大型会议纪要和多人协同。

## 四、语音识别 API 选型

本项目选择阿里云 DashScope Paraformer 实时语音识别。

我对比的方案包括系统语音输入、浏览器 Web Speech API、成熟中文 ASR 服务和 DashScope Paraformer。

| 方案 | 优势 | 不足 |
| --- | --- | --- |
| 系统语音输入 | 入口方便，用户熟悉 | 难以加入本地记忆、指标、声纹过滤和后处理闭环 |
| 浏览器 Web Speech API | 接入简单 | 兼容性和稳定性有限，后端难以统一管理 |
| 传统 ASR 服务 | 中文识别能力成熟 | 接入复杂度、成本和热词机制需要评估 |
| DashScope Paraformer | WebSocket 实时识别、文档清晰、适合后端代理 | 依赖云服务，真实延迟受网络影响 |

选择 DashScope Paraformer 的理由：

- 支持 WebSocket 实时流式识别，适合语音输入场景。
- 后端可以代理请求，避免前端暴露 API Key。
- 支持实时 ASR 和文件转写，便于同时覆盖实时输入和声纹过滤扩展。
- 可以与 OpenAI 兼容 LLM 配置结合，形成 ASR + LLM 后处理链路。
- 接入路径清晰，适合短周期完成可演示 MVP。

## 五、效果评估与持续改善

语音输入法的效果不能只看“是否识别出文字”，还要看“用户是否少改了”“文本是否更可用”。因此我设计了以下评估指标。

### 1. 微整理效果指标

- 冗余词删除率：对“嗯、那个、啊、不对”等样例统计清理效果。
- 自我纠正成功率：测试“不对，是 X”“不是 A，是 B”等表达是否能保留最终意图。
- 用户二次编辑率：微整理后用户是否还需要大量修改。

### 2. 本地记忆效果指标

- 热词学习次数：用户修正后形成了多少本地替换规则。
- 热词命中次数：后续识别中有多少次成功应用替换。
- 平均修改率：用户编辑距离 / 原始文本长度，观察是否随使用次数下降。
- 最近修正记录：用于追踪系统学习是否合理。

### 3. 声纹过滤效果指标

- 说话人数量：文件转写返回的 speaker_count。
- 目标说话人保留比例：过滤后保留的目标说话人文本占比。
- 外来文本残留率：旁人插话是否仍出现在最终输入框。

### 4. 链路稳定性指标

- WebSocket 连接成功率。
- 首字延迟。
- 最终文本返回延迟。
- LLM 处理延迟。
- ASR / LLM 错误次数。

### 5. 已实现的评估入口

前端已实现“输入质量观察”面板，能够展示：

- 累计修正次数。
- 平均修改率。
- 本地热词数量。
- 本地记忆命中次数。
- 最近修正记录。
- 本次处理耗时和处理来源。

这些指标可以让评委直观看到：VoxMem 不是一次性转写工具，而是一个能够通过用户修正持续改进的语音输入系统。

## 六、减少第三方 API 调用次数的思路

第三方 API 调用主要来自 ASR 和 LLM。为了降低成本，我设计了以下策略：

### 1. 原声模式不调用 LLM

如果用户只需要听写和本地记忆，选择原声模式即可。该模式只应用本地替换，不调用 LLM。

### 2. 只对最终文本做处理

实时中间结果变化很快，如果每次都调用 LLM 会造成大量浪费。因此 VoxMem 只在停止录音后，或用户编辑稳定后，对最终文本调用微整理或 Markdown 处理。

### 3. 本地记忆优先

用户已经修正过的个性化词，优先使用 SQLite 本地替换，不必每次都依赖远端热词或 LLM。

### 4. Mock ASR 模式

开发、测试和评审演示可以用 Mock ASR 模式，避免重复消耗真实 ASR 调用。

### 5. 后续缓存策略

后续可以对相同输入、相同模式、相同格式偏好生成缓存，避免重复调用 LLM。

本次已经落地的降本方案包括：Mock ASR、原声模式跳过 LLM、本地记忆优先、只处理最终文本。
