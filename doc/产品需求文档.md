# VoxMem 产品需求与技术规格文档

版本：v0.2
日期：2026-05-23
项目目标：48 小时内完成可演示、可本地部署、可验证核心价值的 MVP。

## 0. 校验结论

原方案的产品方向、核心架构和 48 小时冲刺范围整体成立：VoxMem 应聚焦“会记住用户专有词、并能整理口语表达的极速语音输入”，用实时 ASR + 个性化热词 + 文本后处理 + 修改反馈闭环验证价值。

需要修正的关键点：

1. 阿里云 Paraformer 实时识别支持 WebSocket 双工流式交互，Go 后端可以直接接 WebSocket；但官方 DashScope SDK 当前主要提供 Java/Python，Go 端不应依赖“现成 Go SDK”，而应实现 WebSocket 协议代理。
2. 热词不应描述为“每次连接直接动态注入任意热词数组”。当前应按热词表 ID 使用：`paraformer-realtime-v2` 在 `run-task` 参数里配置 `vocabulary_id`。用户修改后，后端异步更新该用户热词表，通常在下一次识别连接生效。
3. 热词权重和数量要按服务约束设计：单个热词表最多 500 个热词；提升识别概率的权重建议使用 1 到 5。MVP 为降低失败率，后端只同步 Top 50 到 Top 100 高频热词。
4. “今天是星期一，不对，是星期二”这类口头自我纠正，以及“第一点、第二点”这类结构化整理，不应放进热词逻辑。它们属于本次识别结果的文本后处理能力，应在 ASR 最终结果返回后由 LLM 或规则引擎处理。
5. 参考 QuQu 的“ASR 识别 + LLM 文本优化”两段式思路，VoxMem 也应保留原始识别文本和优化后文本两份结果。区别是 VoxMem 是 Web App，LLM API Key 必须放在后端环境变量或服务端配置中，前端不能直接保存或调用第三方模型 Key。

因此，本文件是对原始方案的校正后版本，可以作为 VoxMem MVP 的 PRD + Technical Spec。

## 1. 产品需求分析

### 1.1 用户痛点

系统自带语音输入法的通用准确率已经足够高，但缺少用户级记忆能力。用户反复说人名、公司名、项目名、行业术语时，ASR 常把专有词识别成同音或近音词，例如“张立”被识别成“张力”，“熔断机制”被识别成近似表达。

另一个高频痛点是口语表达本身不等于最终想粘贴或保存的文本。用户会在说话时自我修正，例如“今天是星期一，不对，是星期二”；也会用口头结构表达内容，例如“第一点要确认接口，第二点要补测试”。传统 ASR 往往逐字转写这些过程，导致用户还要手动删除错误片段、补标点、整理成列表或 Markdown。

目标用户需要一个能“越用越懂我、越说越成文”的轻量语音输入工具。它不追求覆盖会议纪要、音频转写、多人协作等大场景，而是优先解决快速输入时的专有词识别、低延迟反馈、口头纠错、结构化输出和可见的自学习效果。

### 1.2 目标用户

- 高频语音输入用户：写备忘、聊天、快速记录、短文本创作。
- 专有名词密集用户：销售、投资、医疗、法律、科研、工程团队。
- 需要结构化输出的用户：写日报、列任务、写邮件、记录 Prompt、整理待办清单。
- 对系统输入法不满意，但不想打开复杂会议转写工具的用户。

### 1.3 同类产品分析

| 产品 | 优势 | 劣势 / VoxMem 机会 |
| --- | --- | --- |
| 系统输入法语音 | 系统级入口，启动成本低 | 无用户级热词记忆；难以量化首字延迟、修改率、热词命中率 |
| 讯飞听见 / 录音笔 | 长音频转写准确率高 | 偏长音频、会议、文件转写；实时输入属性弱 |
| Notta / Otter | 会议记录、多语种、协作能力强 | 功能复杂，离“快速说一句、马上用文本”较远 |
| 通用 ASR Demo | 接入简单，实时返回 | 缺少个人热词闭环和可持续优化机制 |
| QuQu | 采用 ASR + LLM 两段式工作流，强调中文口语优化、自我纠正和文本整理 | 偏桌面端和本地 FunASR；VoxMem 可借鉴文本后处理思路，但保持 Web MVP 和云端实时 ASR 路线 |

### 1.4 核心 KPI

| 指标 | MVP 目标 | 说明 |
| --- | --- | --- |
| 首字延迟 | P50 < 300ms，P90 < 600ms | 从用户开始讲话或音频首帧发送，到前端收到首个有效文本 |
| 最终延迟 | P50 < 800ms | 从用户停止讲话，到前端收到最终句结果 |
| 优化文本延迟 | P50 < 2s，P90 < 4s | 从 ASR 最终文本生成，到后处理文本返回；不计入首字延迟 |
| 热词命中率 | Top 热词命中率逐轮提升 | 对比修正前后的专有词识别结果 |
| 修改率 | 单用户修改次数逐步下降 | 用户编辑距离 / ASR 输出长度 |
| 自我纠正成功率 | MVP 样例集 > 80% | 对“不是 X，是 Y”“不对，是 Y”等明确修正表达的处理准确率 |
| Markdown 整理成功率 | MVP 样例集 > 80% | 对“第一点、第二点、第三点”转列表的格式正确率 |
| 连接成功率 | > 98% | WebSocket 建连、启动任务、完成识别的成功率 |

> 注意：300ms 首字延迟是高压目标，不是云端 ASR 保证值。必须在前端和后端埋点后用真实网络环境验证。文本后处理属于最终文本优化链路，不应拖慢实时出字体验。

## 2. 产品设计与定位

### 2.1 产品定位

VoxMem 是一款“反通用”的极速语音输入工具。

Slogan：Speak it once, remember it forever.

产品原则：

- 快：用户按住就说，文本尽快出现。
- 轻：第一屏就是录音、文本、修改和指标，不做会议工具式复杂入口。
- 会记：用户修正过的词应进入热词库，并在后续识别中改善结果。
- 会整理：用户说出口语化内容后，系统能处理明确自我纠正、口头填充词、基础标点和结构化格式。
- 透明：让用户看到延迟、修改次数、热词命中情况，建立信任。

### 2.2 MVP 核心功能

1. 流式听写
   用户按住或点击录音，浏览器采集麦克风音频，前端通过 WebSocket 发送到 Go 后端，Go 后端代理到阿里云 Paraformer，实时把中间结果和最终结果推回前端。

2. 自进化热词
   用户编辑 ASR 输出后，前端提交原始识别文本和修正文本。后端计算差异、提取候选热词、更新本地热词表，并异步同步到阿里云热词资源。下一次识别时，后端在 `run-task` 中带上该用户的 `vocabulary_id`。

3. 智能文本后处理
   ASR 最终结果返回后，后端对文本做轻量后处理，生成“最终输出文本”。MVP 至少支持轻整理和 Markdown 两种模式：
   - 轻整理：处理明确自我纠正、重复表达、常见口头填充词和基础标点。例如“今天是星期一，不对，是星期二”输出为“今天是星期二”。
   - Markdown：识别“第一点、第二点、第三点”等口头结构，输出 Markdown 列表。例如“第一点确认接口，第二点补测试”输出为有序列表。

4. 数据透明
   前端显示本次首字延迟、最终延迟、优化文本延迟、修改次数、当前热词数。后端记录连接、识别、后处理、修正、热词同步的指标。

### 2.3 MVP 不做范围

- 不做账号注册和多端同步，用户用浏览器 `localStorage` UUID 标识。
- 不做全局输入法 IME 集成，先做 Web App。
- 不做长音频文件转写。
- 不做多人会议、长篇摘要、翻译、协同编辑。
- 不做复杂模板系统。MVP 只做轻整理和 Markdown 两种输出模式。
- 不做移动端原生 App，只声明 Chrome / Edge 优先支持。

## 3. 方案设计

### 3.1 ASR API 选型

选定：阿里云 Paraformer 实时语音识别 WebSocket API。

理由：

- 支持 WebSocket 双工流式传输，适合低延迟输入场景。
- `paraformer-realtime-v2` 支持通过 `vocabulary_id` 使用热词表。
- WebSocket 协议可由 Go 后端直接接入，便于做用户鉴权、热词选择、指标采集和密钥保护。
- 阿里云文档提供清晰的客户端事件、服务端事件和热词管理说明。

关键约束：

- WebSocket 地址：`wss://dashscope.aliyuncs.com/api-ws/v1/inference`。
- 前端不要直接持有阿里云 API Key，必须由后端代理。
- 每次识别任务启动时发送 `run-task` 指令，模型建议使用 `paraformer-realtime-v2`。
- 热词通过 `payload.parameters.vocabulary_id` 传入。热词表更新后不保证影响已启动的识别任务，应按“下次识别生效”设计交互。
- 服务端返回 `result-generated`，其中 `sentence_end=false` 表示中间结果，`sentence_end=true` 表示最终结果。

### 3.2 文本后处理选型

选定：后端代理 OpenAI-compatible Chat Completions API，默认可配置为阿里云 DashScope 兼容模式或其他兼容服务。

理由：

- 口头自我纠正和结构化整理依赖上下文理解，纯规则难以覆盖自然表达。
- OpenAI-compatible API 便于在不同模型服务之间切换，适合 MVP 快速验证。
- 后端代理可以保护 API Key，并统一记录延迟、错误和模型使用情况。
- 参考 QuQu 的两段式工作流，ASR 负责识别，LLM 负责把口语文本转成更可用的最终文本。

关键约束：

- 文本后处理只处理 ASR 最终结果，不处理每个中间结果。
- 前端必须同时保留和展示“原始识别文本”和“最终输出文本”。
- LLM 失败时回退到 ASR 原文，不阻断语音输入。
- 后处理必须保守，禁止添加原文没有的信息。
- 轻整理模式只做明确错误、口头填充、自我纠正和标点处理，不做风格改写。
- Markdown 模式只在用户选择该模式或检测到明显列表结构时启用。
- LLM API Key 不允许出现在前端、日志、浏览器存储或 Git 仓库中。

### 3.3 技术架构

| 层级 | 技术选型 | 理由 |
| --- | --- | --- |
| 前端 | React + TypeScript + Vite | 开发快，状态管理清晰，适合录音、WebSocket、文本编辑和指标展示 |
| 后端 | Go | goroutine 适合双工 WebSocket 代理；部署简单；性能和开发效率平衡 |
| 数据库 | SQLite + GORM | MVP 无需独立数据库服务；本地开发和 Docker 部署简单 |
| ASR | 阿里云 Paraformer WebSocket | 支持实时识别和热词表 ID |
| 文本后处理 | OpenAI-compatible LLM API | 处理自我纠正、口头清理、Markdown 整理；由后端代理调用 |
| 部署 | Docker + Nginx | 单机即可部署，Nginx 负责 TLS、静态资源和 WebSocket 反代 |

### 3.4 部署架构

```text
[ Browser ]
    |
[ Nginx : 443 ]
    |
    +-- static files -> React
    |
    +-- /api/*, /ws/* -> Go API
                         |
                         +-- SQLite
                         |
                         +-- Aliyun Paraformer WebSocket
                         |
                         +-- LLM API (OpenAI-compatible)
```

### 3.5 运行时流程

```text
用户打开页面
  -> 前端生成/读取 user_id
  -> 用户选择输出模式：原声、轻整理或 Markdown
  -> 用户点击录音
  -> 前端连接 /ws/asr?user_id=...
  -> Go 后端读取用户 vocabulary_id 和热词状态
  -> Go 后端连接阿里云 WebSocket
  -> Go 后端发送 run-task，携带 model、format、sample_rate、vocabulary_id
  -> 前端持续发送音频帧
  -> 阿里云返回 result-generated
  -> Go 后端转发文本和指标
  -> 用户停止录音
  -> 后端发送 finish-task 并关闭链路
  -> 后端拿到 ASR 最终文本
  -> 后端按输出模式调用文本后处理
  -> 前端展示原始识别文本和最终输出文本
  -> 用户编辑最终输出文本
  -> 前端 POST /api/correction
  -> 后端保存修正记录，提取热词并标记待同步
  -> 后台任务更新阿里云热词表
```

## 4. 数据模型

### 4.1 SQLite Schema

```sql
CREATE TABLE users (
    id TEXT PRIMARY KEY,
    created_at DATETIME NOT NULL,
    last_seen_at DATETIME NOT NULL
);

CREATE TABLE user_vocabularies (
    user_id TEXT PRIMARY KEY,
    vocabulary_id TEXT,
    sync_status TEXT NOT NULL DEFAULT 'none', -- none, dirty, syncing, synced, failed
    local_version INTEGER NOT NULL DEFAULT 0,
    remote_version INTEGER NOT NULL DEFAULT 0,
    last_synced_at DATETIME,
    last_error TEXT,
    created_at DATETIME NOT NULL,
    updated_at DATETIME NOT NULL
);

CREATE TABLE user_hotwords (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    user_id TEXT NOT NULL,
    word TEXT NOT NULL,
    weight INTEGER NOT NULL DEFAULT 3,
    source TEXT NOT NULL DEFAULT 'correction', -- correction, manual, import
    hit_count INTEGER NOT NULL DEFAULT 0,
    correction_count INTEGER NOT NULL DEFAULT 1,
    last_used_at DATETIME,
    created_at DATETIME NOT NULL,
    updated_at DATETIME NOT NULL,
    UNIQUE(user_id, word)
);

CREATE TABLE corrections (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    user_id TEXT NOT NULL,
    original_text TEXT NOT NULL,
    corrected_text TEXT NOT NULL,
    extracted_words TEXT, -- JSON array
    edit_distance INTEGER NOT NULL DEFAULT 0,
    created_at DATETIME NOT NULL
);

CREATE TABLE asr_sessions (
    id TEXT PRIMARY KEY,
    user_id TEXT NOT NULL,
    vocabulary_id TEXT,
    raw_text TEXT,
    processed_text TEXT,
    processing_mode TEXT,
    first_text_latency_ms INTEGER,
    final_latency_ms INTEGER,
    processing_latency_ms INTEGER,
    status TEXT NOT NULL,
    error_message TEXT,
    started_at DATETIME NOT NULL,
    ended_at DATETIME
);

CREATE TABLE text_processing_jobs (
    id TEXT PRIMARY KEY,
    session_id TEXT NOT NULL,
    user_id TEXT NOT NULL,
    mode TEXT NOT NULL, -- raw, polish, markdown
    input_text TEXT NOT NULL,
    output_text TEXT,
    model TEXT,
    status TEXT NOT NULL, -- pending, processing, completed, failed, skipped
    latency_ms INTEGER,
    error_message TEXT,
    created_at DATETIME NOT NULL,
    completed_at DATETIME
);
```

### 4.2 热词规则

- 只收录用户修正后的“正确词”，例如把“张力”改为“张立”后，收录“张立”。
- 过滤单字、纯标点、过长短语和常见停用词。
- 中文热词长度控制在 2 到 10 个汉字。
- 权重范围使用 1 到 5，默认 3；多次修正或手动添加可提升到 5。
- 同步给阿里云时按 `correction_count`、`hit_count`、`last_used_at` 排序，MVP 只同步 Top 50。

### 4.3 文本后处理规则

输出模式：

| 模式 | 说明 | 示例 |
| --- | --- | --- |
| 原声模式 | 尽量保留 ASR 原文，只做必要标点或不处理 | “今天是星期一不对是星期二”基本保持原样 |
| 轻整理模式 | 处理明确自我纠正、重复、口头填充词和标点 | “今天是星期一，不对，是星期二” -> “今天是星期二” |
| Markdown 模式 | 在轻整理基础上，把明显列表结构整理为 Markdown | “第一点确认接口，第二点补测试” -> 有序列表 |

处理原则：

- 必须保留用户原始意图，禁止添加原文没有的信息。
- 遇到歧义时保守处理，优先保留原文。
- 明确自我纠正只处理强信号表达，例如“不对，是”“不是，是”“改成”“我的意思是”。
- Markdown 模式只处理明显的条目结构，不强行给普通句子加标题。
- 后处理结果必须可编辑，用户修改后的最终文本仍通过 `/api/correction` 进入反馈闭环。
- 后处理不直接生成热词；热词仍来自用户确认或编辑后的修正文本。

## 5. 后端实现计划

### 5.1 服务接口

| 方法 | 路径 | 说明 |
| --- | --- | --- |
| GET | `/healthz` | 健康检查 |
| WS | `/ws/asr` | 前端实时语音连接 |
| POST | `/api/correction` | 提交用户修正文本 |
| POST | `/api/text/process` | 对最终 ASR 文本执行文本后处理 |
| GET | `/api/hotwords` | 查询当前用户热词 |
| POST | `/api/hotwords` | 手动新增热词 |
| DELETE | `/api/hotwords/{id}` | 删除热词 |
| GET | `/api/metrics/summary` | 查询基础指标 |

### 5.2 `/ws/asr` 核心逻辑

1. 校验或创建 `user_id`。
2. 读取 `user_vocabularies.vocabulary_id`。
3. 建立阿里云 WebSocket 连接。
4. 发送 `run-task`，包含：
   - `model: "paraformer-realtime-v2"`
   - `format: "pcm"` 或 `"wav"`
   - `sample_rate: 16000`
   - `vocabulary_id`，仅在已同步成功时传入
   - `semantic_punctuation_enabled: false`，优先降低短句最终延迟
   - `max_sentence_silence` 可按实测调到 300 到 800ms
5. 等待 `task-started` 后开始转发音频。
6. 双工转发：
   - 前端音频二进制帧 -> 阿里云
   - 阿里云 JSON 事件 -> 前端
7. 记录首字延迟、最终延迟、错误信息和 session 状态。
8. 用户停止录音后发送 `finish-task`，等待 `task-finished` 或超时关闭。
9. 保存 ASR 原始最终文本。
10. 如果用户选择轻整理或 Markdown 模式，异步触发文本后处理，并把处理结果推回前端。

### 5.3 `/api/correction` 核心逻辑

请求示例：

```json
{
  "user_id": "browser-generated-uuid",
  "session_id": "asr-session-uuid",
  "original_text": "今天找张力确认一下熔断基质",
  "corrected_text": "今天找张立确认一下熔断机制"
}
```

处理流程：

1. 计算编辑距离和差异片段。
2. 从修正文本中提取候选词。
3. 合并到 `user_hotwords`，更新权重、修正次数和更新时间。
4. 标记 `user_vocabularies.sync_status = dirty`。
5. 返回本次提取结果。
6. 后台同步任务批量更新阿里云热词表，避免每次按键修改都调用远端 API。

### 5.4 `/api/text/process` 核心逻辑

请求示例：

```json
{
  "user_id": "browser-generated-uuid",
  "session_id": "asr-session-uuid",
  "mode": "markdown",
  "input_text": "第一点确认接口第二点补测试第三点更新文档"
}
```

处理流程：

1. 校验 `user_id`、`session_id`、`mode` 和文本长度。
2. 如果 `mode = raw`，直接返回原文或仅做最小标点处理。
3. 如果 `mode = polish`，调用轻整理提示词处理自我纠正、重复和填充词。
4. 如果 `mode = markdown`，调用 Markdown 提示词输出列表或段落结构。
5. 记录 `text_processing_jobs`，包含输入文本、输出文本、模型、状态、延迟和错误。
6. LLM 调用失败时返回原文，并把状态标记为 `failed`，不影响 ASR 主流程。

### 5.5 热词同步任务

同步策略：

- 触发条件：用户提交修正后 2 到 5 秒防抖，或下次识别开始前发现 `dirty`。
- 同步内容：该用户 Top 50 热词。
- 成功后保存 `vocabulary_id`、`remote_version`、`last_synced_at`。
- 失败后保留本地热词，前端提示“热词将在稍后同步”，不阻断语音输入。

### 5.6 文本后处理提示词约束

后端应维护固定的系统提示词模板，不允许前端直接拼接任意系统提示词。

轻整理提示词必须强调：

- 只处理明确的口头错误和自我修正。
- 删除不承载信息的填充词。
- 合并明显重复。
- 保留用户原始语气和用词。
- 不做扩写，不添加事实。

Markdown 提示词必须强调：

- 只在有明显列表、步骤、任务、要点结构时输出 Markdown。
- 优先使用有序列表。
- 不凭空添加标题。
- 不改变条目含义和顺序。
- 输出纯 Markdown，不附带解释。

## 6. 前端实现计划

### 6.1 页面结构

第一屏直接进入工作台：

- 顶部：产品名、连接状态、用户 ID 简短标识。
- 主区：录音按钮、实时识别文本、原始识别文本、可编辑最终输出文本。
- 输出模式：原声、轻整理、Markdown 三段式切换。
- 右侧或底部：首字延迟、最终延迟、优化文本延迟、修改次数、热词数量。
- 热词区：展示 Top 热词，允许手动新增和删除。

### 6.2 关键模块

| 模块 | 说明 |
| --- | --- |
| `useRecorder` | 管理麦克风授权、录音状态、音频采集、PCM 转换 |
| `useASR` | 管理 WebSocket 生命周期、重连、消息解析 |
| `useTextProcessing` | 管理输出模式、后处理请求、原文和最终文本切换 |
| `useCorrection` | 管理文本编辑、diff、修正提交 |
| `LatencyPanel` | 展示首字延迟、最终延迟、连接耗时 |
| `HotwordPanel` | 展示和管理用户热词 |

### 6.3 文本输出交互

- 实时识别过程中优先展示 ASR 中间结果，保证用户立刻看到反馈。
- 停止录音后固定 ASR 原始最终文本。
- 文本后处理进行中时，最终输出区显示处理状态。
- 处理完成后，最终输出区展示轻整理或 Markdown 结果。
- 用户可以一键查看原始识别文本，避免 LLM 改错后无法追溯。
- 用户编辑的是最终输出文本，提交修正时同时带上原始识别文本和最终编辑文本。
- 如果 LLM 失败，最终输出文本回退为 ASR 原文，并显示可重试状态。

### 6.4 音频采集

- 使用 `navigator.mediaDevices.getUserMedia({ audio: true })`。
- 优先使用 `AudioWorklet` 处理音频帧；若 48 小时内实现成本过高，可先用 `MediaRecorder` 验证链路。
- MVP 推荐输出 16kHz 单声道 PCM，以降低后端和 ASR 兼容风险。
- 前端每 50 到 100ms 发送一帧音频，具体大小按实测调整。

## 7. 测试策略

### 7.1 后端单元测试

### 7.3 E2E 测试

工具：Playwright。

用例：

1. 打开页面，Mock 麦克风权限，点击录音，确认页面进入 recording 状态。
2. Mock WebSocket 返回 `result-generated`，确认实时文本更新。
3. 编辑文本并提交，确认 `/api/correction` 被调用。
4. 刷新页面后保留 `user_id`，再次连接时后端尝试使用已同步的 `vocabulary_id`。
5. Mock 最终文本为“今天是星期一，不对，是星期二”，确认轻整理输出“今天是星期二”。
6. Mock 最终文本为“第一点确认接口，第二点补测试”，确认 Markdown 模式输出有序列表。
7. 模拟 LLM 处理失败，确认前端回退 ASR 原文并允许重试。
8. 模拟 ASR 连接失败，确认前端展示错误并允许重试。

## 10. 风险与应对

| 风险 | 影响 | 应对 |
| --- | --- | --- |
| 云端 ASR 首字延迟达不到 300ms | 核心体验变弱 | 用真实指标校准 KPI；优化音频帧大小、网络路径和 VAD 参数 |
| 热词同步不是实时生效 | 用户误以为修改后立即改变 | UI 明确显示“下次识别生效”；后端异步同步并提示状态 |
| 热词 API 配额或失败 | 个性化能力受影响 | 本地热词仍保存；失败重试；限制 Top N 同步 |
| LLM 后处理延迟过高 | 最终文本出现慢 | 实时 ASR 原文先展示；后处理异步返回；记录 `processing_latency_ms` |
| LLM 过度改写或添加内容 | 用户不信任最终文本 | 提示词要求保守；同时保留原始识别文本；用户可一键回退原文 |
| LLM API 不可用或 Key 错误 | 文本整理不可用 | 回退 ASR 原文；前端提示后处理失败，不阻断语音输入 |
| 浏览器音频格式不稳定 | 识别失败或延迟升高 | 优先 Chrome / Edge；统一转 16kHz 单声道 PCM |
| API Key 泄露 | 安全事故 | 阿里云和 LLM API Key 只在后端环境变量；前端只连自有后端 |
| SQLite 并发写入冲突 | 修正提交失败 | Go 后端串行化热词同步，设置 busy timeout |
