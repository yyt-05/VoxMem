# 本地记忆闭环改造方案

## 1. 目标

当前目标不是让用户额外维护“修正记录”，而是把用户自然修改输入框的行为沉淀为本地记忆。

目标流程应调整为：

1. 用户点击录音。
2. ASR 返回原始识别文本。
3. 后端按同一浏览器 `user_id` 应用本地替换映射。
4. 前端输入框显示替换后的可编辑文本。
5. 用户直接在输入框中修改文本。
6. 用户修改输入框后，前端在输入稳定后自动触发 LLM/raw 处理。
7. 后端自动记录 `原始识别文本 -> 用户最终输入文本` 的修正事件。
8. 后端从修正事件里抽取本地替换映射。
9. 学到新映射时弹窗提示，并提供撤销入口。
10. 下次同一浏览器用户再识别到相同错误词时，先本地替换，再进入 LLM。

关键变化：不再要求用户点击“提交修正”，也不要求用户点击处理按钮才触发 LLM。用户只需要完成本来就要做的输入编辑；系统在输入稳定后自动处理和学习。页面主按钮文案统一叫“发送”。

## 2. 当前实现

### 2.1 前端

主要代码在 `web/src/App.tsx`。

当前状态模型：

- `rawFinalText`：ASR 最终原文。
- `enhancedText`：后端应用本地替换后的文本。
- `finalText`：最终输出文本，也是页面 textarea 绑定的值。
- `processedSnapshot`：后端处理完成时的快照，用来判断用户是否修改过。
- `correctionState` / `correctionMessage`：提交修正按钮状态。
- `hotwords`：本地记忆面板展示的替换映射。

当前修正按钮启用条件在 `web/src/App.tsx:170`：

```ts
const canSubmitCorrection =
  rawFinalText.trim() !== '' &&
  finalText.trim() !== '' &&
  finalText.trim() !== processedSnapshot.trim() &&
  correctionState !== 'saving';
```

当前修正提交函数在 `web/src/App.tsx:239`，请求体是：

```ts
{
  user_id: userID,
  session_id: taskId,
  original_text: rawFinalText,
  enhanced_text: enhancedText,
  corrected_text: finalText,
}
```

当前 WebSocket 处理逻辑：

- `transcript` final 消息会更新 `rawFinalText` 和 `liveText`。
- `processing` 消息会更新 `rawFinalText` 和 `enhancedText`。
- `processed` 消息会把后端处理后的文本写入 `finalText`，同时写入 `processedSnapshot`。

当前 UI：

- textarea 标题是“最终输出”。
- 用户改 textarea 后，需要点击“提交修正”按钮。
- “本地记忆”面板展示 `from_text -> to_text`，并允许删除。

### 2.2 后端

主要代码在：

- `server/cmd/server/main.go`
- `server/internal/memory/store.go`
- `server/internal/textproc/processor.go`

当前路由在 `server/cmd/server/main.go:69`：

```go
mux.HandleFunc("GET /ws/asr", asrWebSocketHandler(cfg, logger, store))
mux.HandleFunc("POST /api/correction", correctionHandler(store, logger))
mux.HandleFunc("GET /api/hotwords", hotwordsHandler(store, logger))
mux.HandleFunc("DELETE /api/hotwords/{id}", deleteHotwordHandler(store, logger))
```

当前 WebSocket 收到 ASR 完成后进入 `finalizeOutput`。核心逻辑在 `server/cmd/server/main.go:434`：

1. 裁剪 `originalText`。
2. 调用 `store.ApplyMappings(ctx, userID, originalText)`。
3. 得到 `enhancedText`。
4. 直接调用 `textproc.Process(..., mode, enhancedText)`。
5. 把 LLM 或 raw 处理结果作为 `processed` 消息发给前端。

也就是说，当前顺序是：

```text
ASR 原文 -> 本地替换 -> LLM/raw 处理 -> 前端最终输出 -> 用户编辑 -> 用户点击提交修正
```

SQLite 表在 `server/internal/memory/store.go:72`：

- `users`
- `corrections`
- `hotword_mappings`

当前 `corrections` 表字段：

```sql
id
user_id
session_id
original_text
enhanced_text
corrected_text
created_at
```

当前 `hotword_mappings` 表字段：

```sql
id
user_id
from_text
to_text
correction_count
hit_count
created_at
updated_at
UNIQUE(user_id, from_text, to_text)
```

`SaveCorrection` 在 `server/internal/memory/store.go:128`：

1. 校验 `user_id`、`original_text` 或 `enhanced_text`、`corrected_text`。
2. 插入 `corrections`。
3. 使用 `OriginalText` 作为映射抽取基准；如果为空才使用 `EnhancedText`。
4. 调用 `ExtractMappings(baseText, correctedText)`。
5. upsert 到 `hotword_mappings`。

`ApplyMappings` 在 `server/internal/memory/store.go:182`：

1. 查询当前用户所有映射。
2. 按 `length(from_text) DESC, correction_count DESC, id ASC` 排序。
3. 用 `strings.ReplaceAll` 执行替换。
4. 命中后增加 `hit_count`。

`ExtractMappings` 在 `server/internal/memory/store.go:290`：

- 通过公共前缀和公共后缀找出变化片段。
- 变化片段太短时向外扩展到至少两个 rune。
- 只接受 2 到 16 个 rune 的短文本映射。
- 过滤空值、相同值、纯标点或空白。

## 3. 当前实现的问题

### 3.1 修正动作太刻意

当前用户必须在编辑最终输出后点击“提交修正”。这个动作不是输入流程的一部分，更像为了喂数据而设计的额外按钮。

实际用户预期更像：

```text
录音 -> 出文字 -> 我直接改一下 -> 发送
```

系统应该在用户编辑输入后自动处理和顺手学习，而不是让用户理解并执行“提交修正”。

### 3.2 编辑发生在 LLM 之后

当前前端 textarea 展示的是 `processed` 后的 `finalText`。如果是 `polish` 或 `markdown` 模式，用户编辑的是 LLM 输出，不是进入 LLM 前的输入文本。

这会带来两个问题：

- 从 `rawFinalText -> finalText` 抽取映射时，可能把 LLM 的整理结果也误学成本地替换。
- 用户想修的是识别文本，但页面语义是“最终输出”，心智不一致。

### 3.3 `enhancedText` 已经有字段，但没有成为用户编辑基准

后端已经返回 `enhancedText`，也就是“本地替换后文本”。但当前真正进入 textarea 的是 `processed.text`。

理想情况下，用户应该编辑的是：

```text
ASR 原文应用本地记忆后的文本
```

然后这个用户确认后的文本再进入 LLM。

### 3.4 映射冲突没有处理

当前唯一约束是 `(user_id, from_text, to_text)`，所以同一个用户可以同时存在：

```text
张力 -> 张立
张力 -> 张莉
```

`ApplyMappings` 会按长度、次数、id 顺序替换，但不会明确解决“同一个 from_text 对应多个 to_text”的冲突。新闭环自动化后，这个问题更容易暴露。

## 4. 推荐目标流程

建议把当前链路改成两段：

```text
录音阶段：
ASR 原文 -> 应用本地记忆 -> 前端可编辑输入框

自动处理阶段：
用户最终输入文本 -> debounce 稳定 -> 自动保存修正事件/抽取映射 -> LLM/raw 处理 -> 输出结果
```

更具体地说：

1. `/ws/asr` 只负责录音、ASR、应用本地替换，并把可编辑输入文本返回给前端。
2. 前端 textarea 改成“输入文本”或“待处理文本”，值来自 `enhancedText`。
3. 用户修改 textarea。
4. 前端对输入变更做 debounce，例如 800-1200ms 内没有继续输入后触发处理。
5. 前端调用新的 HTTP 接口，例如 `POST /api/input/commit`。
6. 后端在这个接口里：
   - 保存修正记录。
   - 抽取本地替换映射。
   - 调用 LLM 或 raw 处理。
   - 返回处理结果和本次学到的映射。
7. 前端刷新“本地记忆”面板，并用弹窗提示本次学到的映射，例如“已记录本地记忆：张力 -> 张立”，弹窗内提供撤销。
8. 页面主按钮叫“发送”。它用于发送当前已处理结果到目标位置；LLM 处理本身不依赖点击该按钮。

## 5. 建议修改方案

### 5.1 后端：新增提交入口

建议新增接口：

```text
POST /api/input/commit
```

请求体：

```json
{
  "user_id": "browser-user-id",
  "session_id": "asr-task-id",
  "mode": "polish",
  "original_text": "今天找张力确认方案",
  "enhanced_text": "今天找张力确认方案",
  "final_text": "今天找张立确认方案"
}
```

响应体：

```json
{
  "status": "ok",
  "text": "今天找张立确认方案",
  "mode": "polish",
  "source": "llm",
  "latency_ms": 421,
  "mappings": [
    {
      "id": 1,
      "from_text": "张力",
      "to_text": "张立",
      "correction_count": 1,
      "hit_count": 0
    }
  ]
}
```

处理逻辑：

```go
func inputCommitHandler(store *memory.Store, logger *slog.Logger, cfg config) http.HandlerFunc {
  // 1. decode request
  // 2. validate user_id, final_text, mode
  // 3. if final_text != enhanced_text:
  //      store.SaveCorrection(original_text, enhanced_text, final_text)
  // 4. textproc.Process(ctx, cfg, mode, final_text)
  // 5. return process result and mappings
}
```

说明：

- “自动记录”的触发点应该是输入稳定后的自动处理请求，不是每次键盘输入。
- 不建议在 `onChange` 立即请求后端；前端需要 debounce，并取消或忽略过期请求，避免记录半成品文本。
- 如果 `final_text == enhanced_text`，可以不写 `corrections`，只做 LLM/raw 处理。
- 建议请求体增加 `client_revision` 或 `request_id`，前端只展示最新一次响应，避免慢请求覆盖新输入。

### 5.2 后端：调整 WebSocket 完成事件

当前 `finalizeOutput` 会直接调用 LLM。建议改为只返回可编辑输入：

```text
ASR final -> store.ApplyMappings -> clientEvent{Type: "input_ready", OriginalText, EnhancedText, Text: EnhancedText}
```

也可以先复用现有 `processed` 事件，但为了减少语义混乱，建议引入新事件类型：

```go
Type:         "input_ready",
Text:         enhancedText,
OriginalText: originalText,
EnhancedText: enhancedText,
Mappings:     hits,
```

这样前端可以清楚地区分：

- `transcript`：录音过程中的识别文本。
- `input_ready`：可编辑输入文本已准备好。
- `process_result`：自动处理后的 LLM/raw 输出。

### 5.3 后端：保留现有 `/api/correction`，但不作为主流程

为了兼容当前代码，可以暂时保留 `/api/correction`。但新 UI 不再直接调用它。

推荐演进：

1. 第一版：新增 `/api/input/commit`，内部复用 `store.SaveCorrection` 和 `textproc.Process`。
2. 第二版：确认前端完全迁移后，将 `/api/correction` 标记为 debug/兼容接口。
3. 第三版：如果没有外部调用，再考虑删除。

### 5.4 后端：处理同源映射冲突

建议在 `upsertMapping` 前增加冲突处理。MVP 可以先物理删除同一用户、同一 `from_text`、不同 `to_text` 的旧映射：

```sql
DELETE FROM hotword_mappings
WHERE user_id = ?
  AND from_text = ?
  AND to_text <> ?
```

更稳妥的长期方案是给 `hotword_mappings` 增加：

```sql
enabled INTEGER NOT NULL DEFAULT 1
disabled_at TEXT
disabled_reason TEXT
```

然后 `ApplyMappings` 和 `ListMappings` 只查询 `enabled = 1`。删除按钮也改为软删除。

MVP 可以先不做软删除，但至少要避免同一个 `from_text` 同时指向多个目标。

### 5.5 前端：重命名并重组文本状态

建议把当前 `finalText` 拆成两个概念：

```ts
const [inputText, setInputText] = useState('');
const [inputBaseline, setInputBaseline] = useState('');
const [processedText, setProcessedText] = useState('');
```

含义：

- `inputText`：用户正在编辑、即将提交给 LLM 的文本。
- `inputBaseline`：后端返回的本地替换后文本，用于判断用户是否改过。
- `processedText`：用户提交后，LLM/raw 返回的结果。

原字段可以这样迁移：

| 当前字段 | 新含义 |
| --- | --- |
| `rawFinalText` | 保留，ASR 原始最终文本 |
| `enhancedText` | 保留，后端本地替换后文本 |
| `finalText` | 改为 `inputText` 或拆分 |
| `processedSnapshot` | 可改为 `inputBaseline` |
| `correctionState` | 改为 `commitState` 或 `learningState` |

### 5.6 前端：移除“提交修正”主按钮

当前 UI 在 `web/src/App.tsx:635` 有“提交修正”按钮。建议移除主流程里的这个按钮。

替代方案：

- textarea 所在面板改名为“输入文本”。
- 主按钮叫“发送”。
- 用户输入稳定后自动调用 `/api/input/commit`。
- 输入稳定建议定义为 800-1200ms 内没有继续编辑；新输入出现时取消上一轮请求或忽略旧响应。
- 自动处理过程中展示“处理中”，处理完成后展示 LLM/raw 输出。
- 如果后端返回 `mappings.length > 0`，显示弹窗：

```text
已记录本地记忆：张力 -> 张立
```

弹窗内提供“撤销”按钮，调用现有 `DELETE /api/hotwords/{id}`。如果一次学到多条映射，弹窗可以列出每条映射并分别撤销。

### 5.7 前端：本地记忆面板保留

当前本地记忆面板在 `web/src/App.tsx:654`，可以保留。

建议小改：

- 面板标题仍为“本地记忆”。
- 映射展示仍为 `from_text -> to_text`。
- 删除能力保留。
- 学到新映射时用弹窗提示，不要求用户打开面板才知道系统记住了什么。

### 5.8 测试调整

后端需要新增或调整：

- `inputCommitHandler`：用户修改文本时会保存 correction 并返回映射。
- `inputCommitHandler`：用户没有修改时不写 correction，但仍返回处理结果。
- `inputCommitHandler`：支持重复自动请求时的幂等策略，避免同一输入反复增加 `correction_count`。
- `SaveCorrection`：同一 `from_text` 新目标会覆盖或禁用旧目标。
- `ApplyMappings`：仍然按长词优先替换，并更新 `hit_count`。

现有测试可参考：

- `server/internal/memory/store_test.go`
- `server/cmd/server/main_test.go`

前端 E2E 需要新增：

1. mock ASR 返回“今天找张力确认方案”。
2. 页面显示可编辑输入。
3. 用户改成“今天找张立确认方案”。
4. 等待 debounce 后自动触发 LLM/raw 处理。
5. 页面出现本地记忆弹窗，展示 `张力 -> 张立` 和撤销入口。
6. 点击弹窗撤销后，本地记忆面板不再展示该映射。
7. 不撤销时，再次录音 mock 同一句，输入框直接显示“张立”。

当前 E2E 在 `web/tests/e2e/realtime-asr.spec.ts` 只覆盖录音、停止、最终输出出现 mock 文本。需要扩展到编辑和本地记忆。

## 6. 推荐实施顺序

### 阶段一：后端接口先落地

1. 新增 `POST /api/input/commit`。
2. 复用 `memory.SaveCorrection`。
3. 复用 `textproc.Process`。
4. 增加 handler 测试。

阶段一可以暂时不改 WebSocket，让前端先继续拿 `processed` 文本测试接口。但这只是过渡。

### 阶段二：WebSocket 不再自动 LLM

1. `finalizeOutput` 改为返回 `input_ready`。
2. 不在录音停止后自动调用 `textproc.Process`。
3. 前端收到 `input_ready` 后填充 `inputText`。

这是产品语义真正变正确的一步。

### 阶段三：前端迁移主流程

1. 页面“最终输出”改成“输入文本”。
2. textarea 绑定 `inputText`。
3. 移除“提交修正”按钮。
4. 主按钮文案统一为“发送”。
5. 输入稳定后自动调用 `/api/input/commit`。
6. 返回结果展示到输出区。
7. 返回映射后刷新本地记忆面板，并弹窗提供撤销。

### 阶段四：冲突和撤销

1. 同一 `from_text` 新目标覆盖旧目标。
2. 学习成功弹窗里提供撤销。
3. 如果需要可追溯，再增加软删除字段。

## 7. 最小可行版本

如果只做最小改造，建议范围是：

1. 新增 `/api/input/commit`。
2. 前端移除“提交修正”按钮，主按钮文案改成“发送”。
3. 用户停止编辑 800-1200ms 后自动调用 `/api/input/commit`：
   - 如果用户改过文本，自动保存 correction。
   - 然后调用 LLM/raw 处理。
4. 文案上不再出现“提交修正”。
5. 学到映射时弹窗展示，并提供撤销。

但这个最小版本仍然没有完全解决“编辑发生在 LLM 之后”的问题。它适合快速去掉伪需求感，不适合作为最终闭环。

真正推荐的版本是第 6 节的四阶段方案，尤其是阶段二：先让用户编辑本地替换后的输入，再进入 LLM。

## 8. 已确认的产品决策

本轮已确认三个产品决策：

1. 主按钮叫“发送”。
2. 用户编辑后自动触发 LLM/raw 处理，不要求点击主按钮。
3. 学到新映射时默认弹窗提示，并在弹窗里提供撤销。

实现注意点：

- 自动触发必须有 debounce，不能每个按键都请求后端。
- 前端要处理并发请求，旧响应不能覆盖新输入。
- 后端要避免同一文本被自动请求多次后重复增加修正次数。
