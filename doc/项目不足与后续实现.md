# VoxMem 项目不足与后续实现说明

## 1. 文档目的

本文用于说明 VoxMem 当前版本仍然存在的不足，并重点说明声纹模式在 MVP 阶段的实际实现方式、当前方案的边界，以及后续更完整的声纹识别和目标说话人过滤方案。

VoxMem 当前已经完成实时语音输入、本地记忆、微整理、Markdown 输出和可选声纹过滤链路，但项目仍处于本地可演示 MVP 阶段。当前版本更强调“核心闭环可跑通”，还没有达到生产级语音输入工具的稳定性、实时性和个性化声纹识别能力。

## 2. 当前声纹模式的实现

当前前端提供“去除周围人声”开关。用户开启后，系统不会直接走普通实时 ASR 流程，而是进入一条文件转写和说话人分离链路。

整体流程如下：

```text
浏览器麦克风录音
  -> 前端按音量阈值切分语音片段
  -> 将片段编码为 WAV
  -> POST /api/transcribe/file
  -> 后端上传 WAV 到七牛 Kodo
  -> 后端提交 DashScope 文件转写任务
  -> DashScope 返回 sentences / speaker_id
  -> 前端默认保留第一个 speaker_id 的文本
  -> 用户可以点击“显示全部”恢复所有说话人文本
```

### 2.1 前端处理方式

前端在声纹模式下会持续监听麦克风输入，计算每帧音频的 RMS 和峰值音量。当音量超过阈值时，认为用户开始说话；当静音持续一段时间，或单段语音达到最大时长后，将当前语音片段 flush 出来。

之后前端会把片段合成为 16kHz 单声道 WAV，并调用后端 `/api/transcribe/file` 接口。接口返回后，前端读取 `sentences` 数组里的 `speaker_id`，默认选择第一条句子的说话人作为目标说话人，并拼接该说话人的全部文本。

这种方式的优点是实现简单、演示效果直观，并且能利用文件转写服务返回的说话人分离结果。它不要求实时 ASR 本身支持 speaker_id。

### 2.2 后端处理方式

后端 `/api/transcribe/file` 接口主要做三件事：

1. 接收前端上传的 WAV 音频。
2. 上传到七牛 Kodo，生成 DashScope 可访问的 URL。
3. 调用 DashScope 文件转写接口，并开启说话人分离参数。

文件转写完成后，后端返回：

```json
{
  "sentences": [
    {
      "speaker_id": 1,
      "text": "本人第一句。"
    },
    {
      "speaker_id": 2,
      "text": "旁人插话。"
    }
  ],
  "full_text": "本人第一句。旁人插话。",
  "speaker_count": 2
}
```

前端再基于 `speaker_id` 进行过滤。

### 2.3 与真正声纹识别的区别

当前版本的“声纹过滤”严格来说不是完整的声纹识别。它没有提前注册用户声纹，也没有把实时音频与用户声纹向量做相似度匹配。

当前方案本质是：

```text
说话人分离后处理 + 默认选择第一个说话人
```

真正的声纹识别应当是：

```text
用户注册声纹
  -> 提取用户声纹 embedding
  -> 实时或准实时提取当前音频 embedding
  -> 与目标用户 embedding 比对
  -> 只保留相似度超过阈值的语音
```

因此，当前版本能减少一部分多人串扰，但还不能保证“只识别用户本人”。

## 3. 当前不足

### 3.1 声纹模式不是实时过滤

当前声纹模式需要先录制片段，再上传文件，再等待 DashScope 文件转写任务完成。它和普通实时 ASR 的 WebSocket 出字体验不同，存在明显等待时间。

这意味着当前方案更适合作为“增强过滤模式”，不适合作为真正输入法的默认实时链路。

### 3.2 默认选择第一个说话人存在误判

当前前端默认把第一条 sentence 的 `speaker_id` 当成目标用户。如果旁人先说话，或者录音开头刚好捕捉到环境声，系统可能把旁人误当成目标说话人。

虽然前端提供了“显示全部”的兜底，但还没有提供“选择目标说话人并重新过滤”的完整交互，也没有长期记住用户选择。

### 3.3 没有用户声纹注册

当前系统没有声纹注册流程，例如让用户朗读 5 到 10 秒固定文本来建立个人声纹档案。因此系统无法判断“哪个 speaker_id 是用户本人”，只能根据返回顺序做近似判断。

这也是当前声纹模式最核心的能力缺口。

### 3.4 依赖外部对象存储，配置成本高

DashScope 文件转写需要访问音频 URL，因此后端必须先把 WAV 上传到七牛 Kodo。这样会带来额外配置项，包括 Access Key、Secret Key、Bucket、Region、Domain、私有链接 TTL 等。

对本地评审演示来说，这条链路可接受；但对普通用户部署来说，配置门槛偏高。

### 3.5 音频隐私和生命周期还不完善

当前音频文件上传到对象存储后，主要用于文件转写。后续仍需要补充更严格的生命周期管理，例如转写完成后自动删除对象、限制文件有效期、记录清理日志等。

如果面向真实用户，音频属于敏感数据，不能只依赖临时链接和手动配置。

### 3.6 对重叠说话支持有限

文件转写的说话人分离能处理部分多人交替说话场景，但如果用户和旁人同时说话，仍可能出现一句话中混合多个声音、speaker_id 判断不稳定、文本残留等问题。

这类问题需要更强的语音活动检测、说话人分离模型和置信度策略，当前 MVP 还没有覆盖。

### 3.7 指标评估还不够系统

当前有 `speaker_count`、过滤后文本和全量文本，但还没有形成专门的声纹过滤评估面板。例如：

- 目标说话人保留率。
- 外来文本残留率。
- 用户手动恢复或修改次数。
- 每段文件转写耗时。
- speaker_id 稳定性。

因此当前只能证明链路可用，还不能充分证明过滤效果稳定。

## 4. 后续实现方向

### 4.1 第一阶段：完善当前文件转写方案

短期内可以继续沿用当前“文件转写 + 说话人分离”的方案，但需要补齐交互和可靠性。

计划包括：

1. 在前端展示所有 `speaker_id` 的文本预览，让用户选择“我的声音”。
2. 记录用户本次选择，后续同一段录音中优先保留该 speaker。
3. 增加“过滤后文本 / 全部文本”的对比视图。
4. 对上传音频增加大小限制、时长限制和错误提示。
5. 转写完成后自动清理对象存储文件，减少隐私风险。
6. 在指标面板增加文件转写耗时、speaker_count 和恢复全部次数。

这一阶段的目标是让当前方案更稳定、更可解释，而不是立刻追求完全实时。

### 4.2 第二阶段：增加用户声纹注册

中期应增加用户声纹注册流程。用户首次使用声纹过滤时，系统提示用户朗读一段短文本，收集多段干净音频，提取并保存目标用户的声纹向量。

建议流程：

```text
用户朗读注册文本
  -> 前端录制 5-10 秒音频
  -> 后端做 VAD 去静音
  -> 调用声纹模型提取 embedding
  -> 多段 embedding 求平均
  -> 保存到本地数据库
```

保存内容不应是原始音频，而应优先保存声纹 embedding、模型版本、采样率、创建时间和质量分数。原始注册音频如无必要应尽快删除。

### 4.3 第三阶段：基于 embedding 做目标说话人确认

有了用户声纹 embedding 后，文件转写返回多个 `speaker_id` 时，后端不再让前端默认选择第一个说话人，而是对每个说话人的音频片段提取 embedding，再与用户注册声纹进行相似度计算。

建议流程：

```text
文件转写返回 speaker_id
  -> 按 speaker_id 聚合音频时间段
  -> 对每个 speaker 提取 embedding
  -> 与用户注册 embedding 计算余弦相似度
  -> 选择相似度最高且超过阈值的 speaker
  -> 返回 target_speaker_id / confidence / filtered_text
```

这样可以把当前“默认第一个 speaker”的启发式规则升级为“基于用户声纹匹配”的判断。

### 4.4 第四阶段：准实时声纹门控

如果要兼顾实时输入体验，可以采用准实时方案：前端或后端按 1 到 2 秒窗口切分音频，对每个窗口做 VAD 和声纹 embedding 匹配，只有匹配目标用户的窗口才送入 ASR。

建议流程：

```text
麦克风音频流
  -> VAD 判断是否有人声
  -> 1-2 秒窗口提取 embedding
  -> 与目标用户声纹比对
  -> 通过阈值则送实时 ASR
  -> 未通过则丢弃或降低权重
```

该方案延迟高于纯实时 WebSocket，但低于完整文件转写。它更适合未来的生产版本。

### 4.5 第五阶段：模型和部署选择

后续可以评估两类实现路线：

| 路线 | 优点 | 不足 |
| --- | --- | --- |
| 云端声纹服务 | 接入快，模型维护成本低 | 成本、隐私和厂商能力受限 |
| 本地声纹模型 | 隐私更好，可控性强 | 模型体积、推理延迟和跨平台部署复杂 |

如果选择本地模型，可以优先评估 ECAPA-TDNN、CAM++ 或 FunASR 相关声纹模型。MVP 阶段不建议把模型直接塞进浏览器端，后端推理或独立推理服务会更可控。

## 5. 推荐的后续版本路线

### V0.3：补齐当前声纹过滤体验

- 增加 speaker 选择 UI。
- 增加过滤前后对比。
- 增加音频上传限制和错误提示。
- 增加对象存储自动清理。
- 增加声纹过滤指标。

### V0.4：支持用户声纹注册

- 增加注册引导。
- 保存用户声纹 embedding。
- 增加声纹质量检测。
- 支持重新注册和删除声纹。

### V0.5：目标说话人自动确认

- 按 speaker 聚合音频片段。
- 对每个 speaker 提取 embedding。
- 自动选择目标 speaker。
- 返回匹配置信度和原因。

### V1.0：准实时声纹过滤

- 支持流式 VAD。
- 支持窗口级声纹匹配。
- 在 ASR 前过滤非目标用户音频。
- 降低文件转写依赖，保留文件转写作为高精度兜底。

## 6. 总结

当前 VoxMem 的声纹模式已经完成了可演示的多人声音处理链路：它能够录制音频、调用文件转写、读取 `speaker_id`，并默认过滤为第一个说话人的文本。

但当前方案仍属于 MVP 后处理方案，不是真正的实时声纹识别。它的主要不足是延迟较高、依赖对象存储、默认 speaker 选择可能误判、没有用户声纹注册和 embedding 匹配。

后续更合理的演进路径是：先完善文件转写体验，再增加声纹注册，随后用 embedding 匹配自动确认目标说话人，最后再演进到准实时声纹门控。这样既能保留当前可运行的实现，也能逐步接近真正可用的声纹过滤输入法。
