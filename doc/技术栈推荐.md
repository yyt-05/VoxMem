# VoxMem 技术栈推荐

版本：v0.1
日期：2026-05-23
依据文档：[产品需求文档.md](./产品需求文档.md)

## 1. 推荐结论

VoxMem MVP 推荐采用：

| 层级 | 推荐技术 | 结论 |
| --- | --- | --- |
| 前端 | React + TypeScript + Vite | 保持轻量，适合快速做实时录音、WebSocket 状态和指标面板 |
| 后端 | Go | 作为 WebSocket 双工代理最稳妥，部署简单，性能余量足 |
| ASR | 阿里云 Paraformer Realtime WebSocket | 满足实时识别和 `vocabulary_id` 热词表能力 |
| 数据库 | SQLite + GORM | MVP 单机足够，适合 48 小时内完成闭环 |
| UI | Tailwind CSS + Radix UI + lucide-react | 快速实现清爽工作台，避免从零做交互细节 |
| 测试 | Go testing + Vitest + Playwright | 分别覆盖核心逻辑、前端状态和端到端流程 |
| 部署 | Docker Compose + Nginx | 单机部署，Nginx 负责 TLS、静态资源和 WebSocket 反代 |

核心判断：

- 实时路径必须短：浏览器音频帧 -> Go WebSocket -> 阿里云 WebSocket -> Go -> 浏览器。
- 热词同步不进入实时路径：用户修正后异步同步热词表，下次识别使用 `vocabulary_id`。
- 48 小时内不引入账号系统、消息队列、独立缓存、复杂权限和微服务。

### 2.2 为什么不用更重的前端方案

| 备选 | 不推荐作为 MVP 首选的原因 |
| --- | --- |
| Next.js | SSR、路由和服务端能力对当前 Web App 价值不大，会增加部署复杂度 |
| Electron | 当前 PRD 明确先做 Web App，不做桌面输入法或系统级集成 |
| Angular | 工程体系完整但启动成本较高，不适合 48 小时冲刺 |
| Vue | 可以做，但既有方案已选择 React，生态和测试方案足够成熟 |

### 2.4 前端关键实现建议

- 录音优先用 `AudioWorklet` 输出 16kHz 单声道 PCM。若第一天卡住，可临时用 `MediaRecorder` 打通链路，但最终仍建议回到 PCM。
- WebSocket 消息定义必须用 TypeScript discriminated union，区分 `partial_result`、`final_result`、`metrics`、`error`。
- 文本编辑区保留两份文本：`asrText` 和 `correctedText`。只有用户点击提交或停止编辑后才调用 `/api/correction`。
- 指标面板不要只做前端估算，后端也要返回 session 级指标，便于对齐真实链路延迟。

## 3. 后端技术栈

### 3.1 推荐组合

| 类别 | 推荐 | 版本建议 | 用途 |
| --- | --- | --- | --- |
| 语言 | Go | 1.25+ | WebSocket 双工代理、REST API、后台同步任务 |
| HTTP Router | `github.com/go-chi/chi/v5` | 5.x | 轻量 REST 路由和中间件 |
| WebSocket | `github.com/coder/websocket` | 1.x | 同时处理浏览器连接和阿里云连接 |
| ORM | GORM | 1.x | SQLite 数据模型、CRUD、AutoMigrate |
| SQLite Driver | `github.com/glebarez/sqlite` | 1.x | 纯 Go SQLite，减少 Windows CGO 环境问题 |
| 日志 | `github.com/rs/zerolog` | 1.x | JSON 结构化日志，便于排查 WebSocket 会话 |
| UUID | `github.com/google/uuid` | 1.x | session、用户、本地实体 ID |
| 测试断言 | `github.com/stretchr/testify` | 1.x | 后端单元测试 |
| 文本 Diff | `github.com/sergi/go-diff/diffmatchpatch` | 1.x | 后端提取修正差异 |

本机当前 Go 环境为 `go1.25.4 windows/amd64`，可直接作为 MVP 开发基线。若后续统一 CI，建议锁定 `go 1.25` 或更新到当时的 Go 稳定版，并在 `go.mod` 中明确。

### 3.2 为什么 Go 是后端首选

VoxMem 的后端核心不是传统 CRUD，而是实时双工代理：

```text
Browser WebSocket <-> Go <-> Aliyun Paraformer WebSocket
```

Go 的优势：

- goroutine 适合为每个识别 session 启动读写协程。
- 单二进制部署简单，Docker 镜像小。
- 标准库 HTTP 能力强，不需要重框架。
- 对 WebSocket backpressure、超时、取消和 context 控制更直接。
- 后续迁移到多实例时，Go 服务也容易容器化和水平扩展。

### 3.3 为什么不首选 Python 或 Node 后端

| 备选 | 适合点 | 不作为首选的原因 |
| --- | --- | --- |
| Python + FastAPI | 接 SDK 快，写业务快 | 长连接并发、部署和音频流转发稳定性不如 Go 直接 |
| Node.js + Fastify/NestJS | 前后端同语言，WebSocket 生态成熟 | 音频流代理、后台同步、部署隔离上没有明显超过 Go |
| Rust | 性能强，内存安全 | 48 小时内开发效率偏低，调试成本高 |
| Java / Spring Boot | 稳定，企业级生态强 | 启动和工程复杂度偏高，不适合轻量 MVP |

### 3.5 后端关键实现建议

- `/ws/asr` 每个连接创建一个 `context.Context`，浏览器断开时取消阿里云连接和后台 goroutine。
- 浏览器到后端、后端到阿里云都要设置读写 deadline，避免半开连接拖死 session。
- 阿里云 API Key 只从后端环境变量读取，前端永远不接触。
- 热词同步任务使用单进程内 worker 即可，不上 Redis Queue。
- SQLite 开启 WAL 和 busy timeout：

```sql
PRAGMA journal_mode=WAL;
PRAGMA busy_timeout=5000;
```

## 4. ASR 与热词技术方案

### 4.1 推荐接入方式

实时识别：

- 使用阿里云 Paraformer Realtime WebSocket。
- 模型使用 `paraformer-realtime-v2`。
- 音频格式优先统一为 `pcm`、`sample_rate=16000`、单声道。
- 前端音频分片建议 50 到 100ms 一帧。

热词：

- 后端维护本地 `user_hotwords`。
- 后端异步同步到阿里云热词表。
- 用户识别连接启动时，如果已有同步成功的 `vocabulary_id`，在 `run-task` 参数中传入。
- 用户刚刚修正的词默认下次识别生效，不承诺当前 session 立刻生效。

### 4.2 不建议的方案

| 方案 | 问题 |
| --- | --- |
| 前端直接连阿里云 | API Key 泄露，无法做用户热词和指标控制 |
| 每次识别都重新创建热词表 | 延迟高、容易触发配额或失败，实时路径不稳定 |
| 把全部热词都同步给 ASR | 热词过多会增加错误风险，MVP 只同步 Top 50 |
| 在后端做复杂中文分词 | 48 小时内收益不稳定，先用修正 diff + 手动热词更可控 |

## 5. 数据库与存储

### 5.1 MVP 推荐 SQLite

推荐配置：

| 项 | 推荐 |
| --- | --- |
| 数据库文件 | `data/voxmem.db` |
| ORM | GORM |
| Driver | `github.com/glebarez/sqlite` |
| 迁移 | MVP 用 `AutoMigrate`，上线后改为版本化 migration |
| 写入模式 | WAL |
| 并发策略 | 热词同步串行化，用户修正直接写库 |

SQLite 足以支撑 MVP：

- 当前无账号系统，无多租户隔离需求。
- 用户由浏览器 UUID 标识，数据量小。
- 热词、修正记录、session 指标都是轻量结构化数据。

### 5.2 什么时候升级 PostgreSQL

满足任一条件再升级：

- 需要真实账号体系和跨设备同步。
- 需要多台 Go API 同时写入。
- 需要后台分析大量 session 指标。
- 需要团队级热词库、共享词库、权限控制。

## 6. 部署技术栈

### 6.1 MVP 部署

推荐：

- `Dockerfile.server`：构建 Go API 单二进制。
- `Dockerfile.web`：构建 React 静态资源。
- `nginx.conf`：托管静态资源，反代 `/api/*` 和 `/ws/*`。
- `docker-compose.yml`：本机或单机服务器一键启动。

Nginx 需要支持 WebSocket Upgrade：

```nginx
location /ws/ {
    proxy_pass http://api:8080;
    proxy_http_version 1.1;
    proxy_set_header Upgrade $http_upgrade;
    proxy_set_header Connection "upgrade";
    proxy_set_header Host $host;
}
```

### 6.2 Windows 本地开发命令

前端：

```powershell
cd D:\VoxMem\web
npm install
npm run dev
```

后端：

```powershell
cd D:\VoxMem\server
$env:DASHSCOPE_API_KEY = "your-api-key"
$env:VOXMEM_DB_PATH = "..\data\voxmem.db"
go run .\cmd\api
```

Docker：

```powershell
cd D:\VoxMem
docker compose up --build
```

## 8. 可观测性与指标

MVP 不引入 Prometheus、Grafana、OpenTelemetry。先用结构化日志和数据库指标满足验证。

推荐记录：

| 指标 | 位置 | 用途 |
| --- | --- | --- |
| `first_text_latency_ms` | `asr_sessions` | 验证首字延迟 |
| `final_latency_ms` | `asr_sessions` | 验证最终延迟 |
| `correction_count` | `user_hotwords` | 热词排序 |
| `edit_distance` | `corrections` | 修改率 |
| `sync_status` | `user_vocabularies` | 热词同步健康度 |
| WebSocket error code | 日志 | 排查断连和 ASR 错误 |

上线后再考虑：

- Prometheus `/metrics`
- OpenTelemetry trace
- Sentry 前端错误上报
- Loki / Grafana 日志聚合

## 9. 安全与配置

### 9.1 环境变量

| 变量 | 必填 | 示例 | 说明 |
| --- | --- | --- | --- |
| `DASHSCOPE_API_KEY` | 是 | `sk-...` | 阿里云 DashScope Key |
| `VOXMEM_DB_PATH` | 否 | `.\data\voxmem.db` | SQLite 文件路径 |
| `VOXMEM_ALLOWED_ORIGINS` | 否 | `http://localhost:5173` | CORS 和 WebSocket Origin 白名单 |
| `VOXMEM_ASR_ENDPOINT` | 否 | `wss://dashscope.aliyuncs.com/api-ws/v1/inference` | 便于测试时替换 Fake ASR |
| `VOXMEM_ENV` | 否 | `development` | 环境标识 |

### 9.2 安全底线

- 不把 `DASHSCOPE_API_KEY` 写入前端、Git、README 示例真实值或日志。
- `/ws/asr` 检查 Origin，避免任意站点消耗 ASR 配额。
- 单用户并发识别 session 做限制，MVP 可限制为 1 到 2 个。
- 请求体限制大小，避免 correction API 被大文本打爆。

## 11. 下一步落地顺序

1. 初始化 `web` 和 `server` 两个目录。
2. 先打通 Mock ASR：浏览器 WebSocket -> Go -> Fake ASR -> 浏览器。
3. 接入真实阿里云 Paraformer WebSocket。
4. 实现 correction API 和 SQLite 热词存储。
5. 实现热词表异步同步和 `vocabulary_id` 注入。
6. 增加前端指标面板和热词管理区。
7. 补后端单元测试、前端 Vitest、Playwright Mock E2E。
8. 写 Docker Compose 和 Nginx 配置。

这个技术栈的目标不是“最完整”，而是在 48 小时内把 VoxMem 最关键的价值闭环跑起来：实时出字、用户修正、热词沉淀、下次识别变准。
