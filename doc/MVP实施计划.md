# VoxMem MVP 实施计划

版本：v0.2
日期：2026-05-23
依据文档：
- [产品需求文档.md](./产品需求文档.md)
- [技术栈推荐.md](./技术栈推荐.md)
- [输入界面视觉风格.html](./输入界面视觉风格.html)

## 1. 结论

原实施方案的产品方向是合理的：VoxMem 应该围绕“说话出字快、乱说能整理、改了能记住”这三件事做 MVP。

但原方案不适合作为 48 小时内的执行计划。主要问题是阶段顺序偏重“完整系统”，而不是先拿到一个可演示、可部署、可验证的最小闭环。尤其是阿里云热词 API 同步、完整指标面板、Playwright E2E、AudioWorklet、完整热词 CRUD，都不应该压在第一阶段。

新的实施原则是：

1. 先验证真实 ASR 是否能跑通，再写复杂 UI。
2. 先做最小 Demo，再在 Demo 上加能力。
3. 阶段一只做本地热词替换，不接阿里云热词 API。
4. LLM 后处理是核心 wow 点，必须进入阶段一。
5. 部署是硬性要求，不能留到最后几个小时才处理。
6. 每个阶段都必须有明确的可见产物和验证方法。

## 2. 原方案合理性评估

### 2.1 可以保留的部分

- React + TypeScript + Vite 作为前端。
- Go 作为 WebSocket 代理和后端 API。
- SQLite 作为 MVP 本地数据库。
- Docker + Nginx 作为单机部署方案。
- 浏览器录音到 Go 后端，再由 Go 后端代理阿里云 Paraformer。
- 用户修正后形成个人热词记忆。
- 保留 Mock ASR，但只用于前端调试和自动化测试，不作为 MVP 成功标准。

### 2.2 必须调整的部分

| 原方案点 | 问题 | 调整 |
| --- | --- | --- |
| 先做 Mock ASR，再接真实 ASR | Mock 不能验证最关键的音频格式和阿里云协议 | 第 0 阶段先用预录 PCM 验证真实阿里云 ASR |
| 阶段一接阿里云热词 API | 异步生效、调试复杂、失败概率高 | 阶段一只做本地 `from_text -> to_text` 替换 |
| 以 AudioWorklet 为优先方案 | 48 小时内调试风险高 | 阶段一优先 `AudioContext + ScriptProcessorNode`，Chrome / Edge 优先 |
| 指标面板、E2E、完整 CRUD 过早 | 占用主链路时间 | 阶段一只保留最小指标和手动验证 |
| LLM 后处理不是第一优先级 | 与产品需求不一致 | 阶段一必须支持自我纠正和 Markdown 整理 |
| 部署排在最后 | 评委必须亲手体验，部署不能冒险 | 核心链路跑通后立即做 Docker Compose |

## 3. MVP 成功标准

阶段一 MVP 只用评委 3 到 5 分钟体验路径验收：

1. 评委打开浏览器，看到简洁输入工作台。
2. 点击录音，说一句短话，页面实时出字。
3. 说“今天是星期一，不对今天是星期二”，最终整理文本显示“今天是星期二”。
4. 切换 Markdown 模式，说“第一点确认接口，第二点补测试”，最终输出 Markdown 列表。
5. 故意让系统识别错一个词，手动改正并提交。
6. 再次说包含该错误词的句子，系统通过本地热词替换显示正确词。
7. 用 Docker 部署地址完成同一条路径。

只要以上路径稳定，就认为阶段一成功。

阶段一不以“功能列表完整”为成功标准，不追求完整账户系统、复杂后台、阿里云热词同步、完整测试覆盖或多端兼容。

## 4. 总体阶段规划

| 阶段 | 时间建议 | 目标 | 成功产物 |
| --- | --- | --- | --- |
| 阶段 0 | 0-1h | 真实 ASR 技术探针 | 预录 16k PCM 能被阿里云识别 |
| 阶段 1 | 1-8h | 最小实时出字 Demo | 浏览器录音后实时看到文字 |
| 阶段 2 | 8-14h | LLM 文本整理 Demo | 自我纠正和 Markdown 输出可用 |
| 阶段 3 | 14-22h | 本地记忆 Demo | 修改一次后，下次本地替换生效 |
| 阶段 4 | 22-30h | 可部署 Demo | Docker Compose 可启动完整服务 |
| 阶段 5 | 30-40h | 体验打磨 | 错误态、空态、loading、演示路径稳定 |
| 阶段 6 | 40-48h | 最终验收和交付 | README、演示脚本、部署验证、兜底方案 |

## 5. 阶段 0：真实 ASR 技术探针

目标：在写前端录音和复杂后端之前，先证明阿里云 Paraformer 链路可用。

这是整个项目最关键的正确性检查。如果这一步过不了，后续写再多 UI 都没有意义。

### 0.1 准备预录音频样本

操作：
- 准备一段 3 到 5 秒中文语音。
- 转成 16kHz、单声道、PCM 格式。
- 语音内容使用简单短句，例如“今天下午确认接口方案”。

验证方法：
- 使用本地播放器或音频工具确认音频不是静音。
- 确认采样率是 16kHz。
- 确认声道是单声道。
- 确认格式是阿里云实时识别可接受的 PCM 或 WAV。

完成标准：
- 本地有一个明确可复用的测试音频文件。
- 音频格式信息可被检查，不靠猜测。

### 0.2 验证阿里云 WebSocket 连接

操作：
- 后端或临时探针连接阿里云 Paraformer WebSocket。
- 使用后端环境变量读取 API Key。
- 发送 `run-task` 事件。
- 等待 `task-started`。

验证方法：
- Key 正确时能收到任务启动成功事件。
- Key 错误时能收到明确失败信息。
- 日志中不出现完整 API Key。

完成标准：
- 能确认鉴权、WebSocket 地址、模型名称和任务启动事件都正确。

### 0.3 发送预录 PCM 并拿到文字

操作：
- 按固定帧大小发送预录 PCM 音频。
- 发送结束事件。
- 等待最终识别文本。

验证方法：
- 识别结果中能看到预期中文内容。
- 中间结果和最终结果能区分。
- 停止发送后连接能正常关闭。

完成标准：
- 预录 PCM 到阿里云再到文本的链路跑通。
- 如果 1 小时内仍不能跑通，不进入复杂前端开发，先修 ASR 协议和音频格式。

## 6. 阶段 1：最小实时出字 Demo

目标：用户在浏览器点录音，说话后能实时看到文字。

这一阶段只做最小工作台，不做数据库、不做 LLM、不做热词、不做复杂 UI。

### 1.1 初始化最小工程结构

操作：
- 保留现有 `doc` 目录。
- 创建 `web`、`server`、`data`、`deploy` 目录。
- 前端初始化 React + TypeScript + Vite。
- 后端初始化 Go 服务。
- 后端提供健康检查接口。

验证方法：
- 前端开发服务能打开页面。
- 后端健康检查能返回成功。
- 前端能请求后端健康检查。
- 根目录结构清晰，数据库文件不会被提交。

完成标准：
- 前后端能同时本地启动。
- 页面不是空白。
- 后端错误能在终端看到。

### 1.2 实现最小输入工作台

操作：
- 页面第一屏直接显示输入工作台。
- 放置录音按钮。
- 放置实时识别文本区。
- 放置最终识别文本区。
- 放置连接状态。
- 放置错误提示区域。

验证方法：
- 打开页面后不需要跳转即可看到录音入口。
- 点击按钮能切换状态。
- 没有后端连接时能显示错误。

完成标准：
- 评委第一眼知道可以点击录音。
- 页面没有营销页或无关介绍。

### 1.3 实现浏览器录音采集

操作：
- 使用浏览器麦克风权限。
- 阶段一优先使用 `AudioContext + ScriptProcessorNode`。
- 将音频转换为 16kHz、单声道、PCM。
- 每 50 到 100ms 向后端发送一帧音频。
- 停止录音后停止发送音频帧。

验证方法：
- 首次点击录音时浏览器弹出麦克风权限。
- 拒绝权限后页面不崩溃，并能重试。
- 授权后后端能持续收到音频帧。
- 停止录音后后端不再收到新音频帧。

完成标准：
- Chrome / Edge 上音频采集稳定。
- 连续录音 30 秒页面不明显卡顿。

### 1.4 实现前后端 WebSocket

操作：
- 前端点击录音时连接后端 `/ws/asr`。
- 后端接收连接并记录 session。
- 前端发送音频帧。
- 后端收到音频帧后转发给 ASR 连接。
- 后端收到 ASR 结果后转发给前端。

验证方法：
- 前端能建立 WebSocket。
- 后端能识别连接开始、音频帧、停止事件。
- 后端断开时前端能显示错误并允许重试。

完成标准：
- 录音期间前后端连接状态准确。
- 一次录音结束后可以再次录音。

### 1.5 接入真实阿里云 ASR

操作：
- 后端建立到阿里云 Paraformer 的 WebSocket 连接。
- 后端发送任务启动事件。
- 任务启动成功后再转发浏览器音频。
- 后端把中间结果和最终结果推回前端。
- 结束录音时发送结束事件并关闭连接。

验证方法：
- 用户说短句时，前端能看到实时文字出现。
- 停止录音后能收到最终文本。
- 刷新页面后可再次录音。
- API Key 不出现在前端代码、浏览器控制台和普通日志中。

完成标准：
- 至少连续完成 5 次短句识别。
- 每次都能从录音状态回到可再次录音状态。
- 如果这一阶段到第 8 小时还没有跑通，必须暂停 LLM 和热词开发，继续修音频链路。

## 7. 阶段 2：LLM 文本整理 Demo

目标：ASR 最终文本出来后，系统能把口语表达整理成可用文本。

这一阶段实现两个核心需求：
- “今天是星期一，不对今天是星期二”应输出“今天是星期二”。
- 用户说一段列表时，可以整理成 Markdown。

### 2.1 增加输出模式

操作：
- 前端增加输出模式切换。
- 阶段一只保留三个模式：原声、轻整理、Markdown。
- 默认使用轻整理。
- 实时识别区仍然显示 ASR 原始文本。
- 最终输出区显示 LLM 整理后的文本。

验证方法：
- 切换模式不会影响实时出字。
- 停止录音后才触发文本整理。
- 原声模式不调用 LLM。

完成标准：
- 用户能明确看到“实时识别文本”和“最终输出文本”的区别。

### 2.2 实现后端 LLM 调用

操作：
- 后端通过 OpenAI-compatible API 调用 LLM。
- API Key 只放在后端环境变量。
- 给 LLM 设置超时。
- LLM 调用失败时返回 ASR 最终文本。

验证方法：
- Key 正确时能返回整理文本。
- Key 错误或超时时，页面显示原文并提示后处理失败。
- LLM 失败不影响下一次录音。

完成标准：
- 文本后处理是异步增强层，不阻塞实时出字。

### 2.3 打磨轻整理提示词

操作：
- 提示词要求保留用户原意。
- 提示词要求处理明确自我纠正表达。
- 提示词要求删除明显口头填充词和重复片段。
- 提示词禁止扩写原文没有的信息。

验证方法：
- 输入“今天是星期一，不对今天是星期二”，输出“今天是星期二”。
- 输入“不对，应该是明天下午三点”，输出保留修正后的时间。
- 输入含口头词的长句，输出更清晰但不改变事实。

完成标准：
- 自我纠正样例集通过率达到可演示水平。
- 输出稳定，不出现解释性废话。

### 2.4 打磨 Markdown 提示词

操作：
- Markdown 模式只输出 Markdown 正文。
- 明确识别“第一点、第二点、第三点”。
- 明确识别“首先、然后、最后”。
- 保持用户原始信息，不额外补充内容。

验证方法：
- 输入“第一点确认接口，第二点补测试”，输出有序列表。
- 输入“今天要做三件事，确认接口，补测试，写 README”，输出列表。
- 输入普通短句时不强行生成复杂结构。

完成标准：
- 评委说一段列表时，最终输出能明显变成结构化 Markdown。

## 8. 阶段 3：本地记忆 Demo

目标：用户改过一次后，下次能看到“它记住了”的效果。

阶段一不接阿里云热词 API。只做本地替换层，保证 Demo 稳定、立即生效、可解释。

### 3.1 建立 SQLite 最小数据表

操作：
- 建立匿名用户表。
- 建立识别 session 表。
- 建立修正记录表。
- 建立本地热词映射表，保存 `from_text -> to_text`。
- 建立热词展示表，保存用户看到的正确词。

验证方法：
- 后端启动时数据库文件能自动创建。
- 重启后数据不丢失。
- 同一浏览器刷新后仍然使用同一个用户标识。

完成标准：
- 数据库只承担阶段一必要闭环，不提前扩成复杂后台。

### 3.2 实现修正提交

操作：
- 前端允许用户编辑最终输出文本。
- 用户点击提交修正。
- 请求同时携带原始识别文本、增强后文本和用户最终修正文。
- 后端保存修正记录。

验证方法：
- 把“张力”改成“张立”后，数据库出现一条修正记录。
- 没有实际修改时，不提示用户提交修正。
- 修正失败时页面可重试。

完成标准：
- 用户修正行为可持久化。

### 3.3 提取本地替换映射

操作：
- 从原文和修正文中提取候选替换对。
- 优先保存明确短语替换，例如“张力 -> 张立”。
- 过滤单字、纯标点、过长短语。
- 同一用户同一替换对重复提交时增加计数，不重复插入。

验证方法：
- “今天找张力确认方案”改为“今天找张立确认方案”，生成“张力 -> 张立”。
- “熔断基质”改为“熔断机制”，生成“熔断基质 -> 熔断机制”。
- 空修改不生成映射。

完成标准：
- 本地映射可解释、可查看、可删除。

### 3.4 在识别结果中应用本地替换

操作：
- ASR 最终文本返回后，先应用本地替换。
- 再进入 LLM 后处理。
- 多个映射命中时，优先替换更长的 `from_text`。
- 记录本次命中的替换项。

验证方法：
- 第一次识别出“张力”后，用户改成“张立”并提交。
- 第二次 ASR 最终文本中再次出现“张力”时，最终输出变成“张立”。
- 本地替换发生在 LLM 之前，LLM 看到的是增强后的文本。

完成标准：
- 评委能在一次演示中看到“改了能记住”。

### 3.5 增加最小热词面板

操作：
- 展示最近生成的本地替换对。
- 展示正确词和修正次数。
- 允许删除错误替换。
- 不做复杂搜索、批量导入、远程同步状态。

验证方法：
- 提交修正后面板刷新。
- 删除替换后，下次不再应用该替换。
- 刷新页面后列表仍然存在。

完成标准：
- 用户能理解系统记住了什么，也能撤销错误记忆。

## 9. 阶段 4：可部署 Demo

目标：不要只在开发机上跑通，要让评委可以通过部署地址体验。

### 4.1 整理环境变量

操作：
- 明确后端需要的 ASR API Key。
- 明确后端需要的 LLM API Key、Base URL 和模型名。
- 明确 SQLite 数据库路径。
- 明确允许访问的前端 Origin。
- 提供示例配置，不提交真实 Key。

验证方法：
- 不配置 Key 时，后端给出明确错误。
- 配置 Key 后，ASR 和 LLM 都能调用。
- 前端构建产物不包含任何真实 Key。

完成标准：
- 部署者知道要配置什么。
- 密钥不会泄露到浏览器。

### 4.2 容器化后端

操作：
- 构建 Go 后端运行镜像。
- 容器启动后读取环境变量。
- 容器内使用挂载目录保存 SQLite。
- 容器启动失败时输出明确日志。

验证方法：
- 容器启动后健康检查通过。
- 重启容器后数据库仍然存在。
- 日志中不出现完整 Key。

完成标准：
- 后端可以脱离本地开发命令运行。

### 4.3 构建前端静态资源

操作：
- 构建前端生产包。
- 使用 Nginx 托管静态资源。
- 前端通过同源路径访问后端 API 和 WebSocket。

验证方法：
- 生产页面能打开。
- 刷新页面不 404。
- API 和 WebSocket 都能通过 Nginx 访问。

完成标准：
- 用户只需要打开一个地址。

### 4.4 配置 Docker Compose

操作：
- 使用 Docker Compose 编排 Nginx 和 Go API。
- 挂载 `data` 目录。
- 注入环境变量。
- 暴露一个浏览器访问端口。

验证方法：
- 一条启动命令后页面可访问。
- 能完整完成录音、识别、LLM 整理和修正提交。
- 重建容器后热词映射不丢失。

完成标准：
- 部署路径可复现。
- 不是只在开发服务下可用。

## 10. 阶段 5：体验打磨

目标：让评委体验路径顺，不因为小错误中断。

### 5.1 打磨状态机

操作：
- 明确空闲、连接中、录音中、停止中、处理中、完成、错误状态。
- 按状态控制录音按钮可用性。
- 错误后允许重新开始。

验证方法：
- 正常录音状态流转正确。

## 14. 风险和应对

| 风险 | 影响 | 应对 |
| --- | --- | --- |
| 真实 ASR 探针失败 | 项目核心链路不成立 | 阶段 0 阻断后续开发，先修 Key、协议、音频格式 |
| 浏览器 PCM 格式不对 | 实时识别乱码或无结果 | 用预录 PCM 对照；浏览器输出统一转 16kHz 单声道 |
| ASR 首字延迟不达预期 | 第一印象变弱 | 先保证稳定出字，再调帧大小、静音参数和网络路径 |
| LLM 延迟高 | 最终文本出现慢 | 原始 ASR 先显示；LLM 异步返回；失败回退原文 |
| LLM 过度改写 | 用户不信任结果 | 提示词保守；保留原始识别文本；用户可编辑 |
| 本地替换误伤 | 文本被错误替换 | 只保存明确修正对；长词优先；允许删除 |
| Docker 部署失败 | 评委无法体验 | 核心链路后立即做部署，不压到最后 |
| SQLite 数据丢失 | 热词记忆失效 | Docker 挂载 `data` 目录；验收时重建容器验证持久化 |
| API Key 泄露 | 安全问题 | Key 只在后端环境变量；日志脱敏；前端不接触 Key |

## 15. 最小 Demo 验收清单

### 15.1 ASR 链路

- 连续跑 5 遍没有阻断性错误。
- README 能指导别人启动和体验。

## 16. 最终判断

按原方案完整做，48 小时内成功率不高，因为范围太大，而且风险排序不对。

按本计划裁剪后，MVP 可以实现。关键前提是第 0 阶段和第 1 阶段必须尽早跑通真实 ASR。如果真实 ASR 链路在前 8 小时内完成，后续 LLM 整理、本地热词替换、SQLite 持久化和 Docker 部署都属于可控工程工作。

阶段一最重要的取舍是：不要追求完整产品，只追求评委能亲手体验到三件事。

- 说话出字快。
- 乱说能整理。
- 改了能记住。
