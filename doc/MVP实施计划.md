## 1. 文档目标

本文档用于指导 VoxMem MVP 的具体落地实施。

目标是把“会记住用户专有词的极速语音输入”做成一个可本地运行、可演示、可验证核心价值的 Web App。

本文档只描述实施步骤、验证方法、完成标准和风险处理，不包含业务代码。

## 2. MVP 范围

### 2.1 必须完成

- 浏览器打开后直接进入语音输入工作台。
- 浏览器生成并持久化匿名 `user_id`。
- 用户可以点击录音，前端通过 WebSocket 连接 Go 后端。
- Go 后端作为 WebSocket 双工代理，连接阿里云 Paraformer 实时识别服务。
- 前端可以实时展示中间识别文本和最终识别文本。
- 用户可以编辑识别结果并提交修正。
- 后端可以根据修正记录提取用户热词。
- 后端可以把 Top 热词异步同步到阿里云热词表。
- 下一次识别时，如果热词表已同步成功，后端在识别任务中带上 `vocabulary_id`。
- 前端展示首字延迟、最终延迟、修改次数、热词数量和热词同步状态。
- 提供 Mock ASR 模式，保证没有阿里云 Key 时也能完成端到端演示和自动化测试。
- 提供本地开发启动方式和 Docker Compose 部署方式。

### 2.2 MVP 不做

- 不做账号注册和登录。
- 不做多端同步。
- 不做系统输入法 IME 集成。
- 不做长音频文件转写。
- 不做会议纪要、摘要、翻译、多人协作。
- 不做移动端原生 App。
- 不做复杂后台管理系统。
- 不做多机分布式部署。

## 3. 关键技术决策

### 3.1 数据库决策

MVP 使用 SQLite。

原因：

- 部署简单，不需要单独安装数据库服务。
- 数据库就是一个文件，适合本地演示和单机部署。
- Docker 部署时只需要持久化 `data` 目录。
- 当前只有匿名用户、热词、修正记录和 session 指标，数据规模小。

推荐数据库文件路径：

- 本地开发：`D:\VoxMem\data\voxmem.db`
- Docker 容器内：`/app/data/voxmem.db`

部署时必须保证 `data` 目录是持久化目录。容器重建后，`voxmem.db` 不能丢失。

暂不使用 PostgreSQL。只有出现以下情况时再升级：

- 需要真实账号系统。
- 需要跨设备同步。
- 需要多台后端实例同时写入。
- 需要团队级共享热词库。
- 需要大规模数据分析。

### 3.2 ASR 接入决策

实时识别使用阿里云 Paraformer Realtime WebSocket。

落地顺序必须是：

1. 先实现 Mock ASR。
2. 再接真实阿里云 WebSocket。
3. 最后接热词表同步和 `vocabulary_id` 注入。

这样可以避免第一天就被云端鉴权、音频格式或网络问题卡死。

### 3.3 热词同步决策

热词同步不进入实时路径。

用户提交修正后：

1. 后端立即保存本地修正和本地热词。
2. 后端把该用户热词表标记为 `dirty`。
3. 后台任务异步同步 Top 50 热词到阿里云。
4. 同步成功后保存 `vocabulary_id`。
5. 下一次识别连接启动时使用该 `vocabulary_id`。

前端必须明确表达“热词下次识别生效”，不能暗示当前录音会立即变准。

## 4. 目标目录结构

实施完成后，仓库建议结构如下：

```text
D:\VoxMem
  doc\
  web\
  server\
  data\
  deploy\
  README.md
```

目录职责：

| 目录 | 职责 |
| --- | --- |
| `doc` | 产品文档、技术选型、实施计划、项目流程文档 |
| `web` | React + TypeScript + Vite 前端应用 |
| `server` | Go 后端服务、WebSocket 代理、REST API、SQLite 数据访问 |
| `data` | 本地 SQLite 数据库文件，部署时需要持久化 |
| `deploy` | Dockerfile、Nginx 配置、Docker Compose 配置 |

## 6. 阶段 0：工程准备

### 0.1 确认开发环境

操作：

- 确认 Windows PowerShell 可用。
- 确认 Go 已安装。
- 确认 Node.js 和 npm 已安装。
- 确认 Docker Desktop 已安装，后续部署验证会用到。

验证方法：

- 在 PowerShell 中检查 Go 版本。
- 在 PowerShell 中检查 Node.js 和 npm 版本。
- 在 PowerShell 中检查 Docker 版本。

完成标准：

- Go、Node.js、npm、Docker 都能在 PowerShell 中正常执行。

### 0.2 建立目录

操作：

- 在仓库根目录创建 `web`、`server`、`data`、`deploy` 目录。
- 保留现有 `doc` 目录。
- 确认 `data` 目录不会误提交数据库文件。

验证方法：

- 在 PowerShell 中列出仓库根目录。
- 确认新目录存在。
- 确认 `.gitignore` 已覆盖 SQLite 数据库文件。

完成标准：

- 目录结构清晰。
- 数据文件不会被误提交到 Git。

### 0.3 定义环境变量

操作：

- 定义后端需要读取的环境变量。
- 至少包含 `DASHSCOPE_API_KEY`、`VOXMEM_DB_PATH`、`VOXMEM_ALLOWED_ORIGINS`、`VOXMEM_ASR_ENDPOINT`、`VOXMEM_ENV`。
- README 中只写示例，不写真实 Key。

验证方法：

- 不设置 `DASHSCOPE_API_KEY` 时，Mock 模式仍可运行。
- 设置真实 Key 后，真实 ASR 模式可运行。

完成标准：

- 前端永远不接触阿里云 Key。
- Key 不出现在 Git 历史、前端构建产物、浏览器控制台和日志中。

## 7. 阶段 1：初始化前后端骨架

### 1.1 初始化前端应用

操作：

- 在 `web` 目录初始化 React + TypeScript + Vite 项目。
- 安装 UI、状态、请求、测试相关依赖。
- 先保留一个最小页面，确认构建链路正常。

验证方法：

- 在 PowerShell 中进入 `D:\VoxMem\web`。
- 启动前端开发服务。
- 浏览器访问本地开发地址。

完成标准：

- 页面能打开。
- 浏览器控制台无启动错误。
- 前端热更新可用。

### 1.2 初始化后端应用

操作：

- 在 `server` 目录初始化 Go module。
- 建立后端入口目录。
- 实现最小 HTTP 服务。
- 提供 `/healthz` 健康检查。

验证方法：

- 在 PowerShell 中进入 `D:\VoxMem\server`。
- 启动 Go 服务。
- 访问 `/healthz`。

完成标准：

- `/healthz` 返回成功状态。
- 服务启动失败时能看到明确错误日志。

### 1.3 配置前后端本地联调

操作：

- 前端通过配置访问后端 API 地址。
- 后端允许本地前端开发源访问。
- 先只允许明确的本地 Origin，不开放任意 Origin。

验证方法：

- 前端页面请求 `/healthz` 或等价健康检查接口。
- 后端日志能看到请求。

完成标准：

- 前端能访问后端。
- CORS 或 Origin 校验不会误拦截本地开发。

## 8. 阶段 2：数据库和基础 API

### 2.1 建立 SQLite 连接

操作：

- 后端启动时根据 `VOXMEM_DB_PATH` 打开 SQLite。
- 如果目录不存在，启动时创建目录。
- 配置 WAL。
- 配置 busy timeout。

验证方法：

- 删除本地数据库文件后启动服务。
- 确认 `data\voxmem.db` 自动创建。
- 查看日志确认数据库初始化成功。

完成标准：

- 数据库文件可自动创建。
- 后端重启后数据仍保留。

### 2.2 建立数据表

操作：

- 建立 `users` 表。
- 建立 `user_vocabularies` 表。
- 建立 `user_hotwords` 表。
- 建立 `corrections` 表。
- 建立 `asr_sessions` 表。

验证方法：

- 启动服务后检查数据库表是否存在。
- 重复启动服务不会重复创建冲突。

完成标准：

- 表结构覆盖 PRD 中定义的数据。
- 所有时间字段、状态字段、唯一约束都符合后续业务需要。

### 2.3 实现用户识别

操作：

- 前端首次打开页面时生成浏览器匿名 `user_id`。
- 前端把 `user_id` 带到 API 和 WebSocket 请求中。
- 后端收到 `user_id` 后创建或更新用户记录。

验证方法：

- 首次打开页面后数据库出现用户记录。
- 刷新页面后 `user_id` 不变化。
- 清空浏览器本地存储后重新生成新用户。

完成标准：

- 匿名用户链路稳定。
- 不需要注册登录即可使用。

### 2.4 实现热词查询 API

操作：

- 后端提供当前用户热词列表查询。
- 返回热词、权重、命中次数、修正次数、同步状态等必要信息。
- 前端热词面板消费该接口。

验证方法：

- 新用户热词列表为空。
- 手动插入测试热词后，前端能显示。

完成标准：

- 热词列表不使用硬编码数据。
- 空状态显示自然，不报错。

### 2.5 实现手动新增热词 API

操作：

- 前端提供新增热词入口。
- 后端校验热词长度、内容和重复项。
- 新增成功后标记用户热词表为 `dirty`。

验证方法：

- 新增“张立”成功。
- 重复新增不会生成两条相同热词。
- 新增单字、空字符串、纯标点时被拒绝。

完成标准：

- 手动热词可用。
- 数据库状态正确更新。

### 2.6 实现删除热词 API

操作：

- 前端提供删除入口。
- 后端按热词 ID 删除当前用户自己的热词。
- 删除后标记用户热词表为 `dirty`。

验证方法：

- 删除后列表立即刷新。
- 刷新页面后被删除热词不再出现。
- 用户不能删除其他用户的热词。

完成标准：

- 删除行为可靠。
- 删除后远端热词表后续会被重新同步。

## 9. 阶段 3：Mock ASR 实时链路

### 3.1 定义前后端 WebSocket 消息类型

操作：

- 明确前端发给后端的消息类型。
- 明确后端推给前端的消息类型。
- 至少覆盖连接成功、中间结果、最终结果、指标、错误、结束。

验证方法：

- 前端收到未知消息时不会崩溃。
- 后端收到非法消息时能返回明确错误或关闭连接。

完成标准：

- WebSocket 消息协议稳定。
- 前后端不依赖临时字段。

### 3.2 实现后端 `/ws/asr`

操作：

- 后端提供 `/ws/asr`。
- 连接时校验 `user_id`。
- 创建 ASR session。
- 记录开始时间、用户 ID、状态。

验证方法：

- 前端连接 WebSocket 后，数据库出现 session。
- 断开连接后 session 状态更新。

完成标准：

- 每次录音都有可追踪 session。

### 3.3 实现 Mock ASR 服务逻辑

操作：

- 后端提供 Mock ASR 模式。
- Mock 模式下不连接阿里云。
- 后端收到音频帧后，按固定节奏返回模拟中间结果和最终结果。

验证方法：

- 不配置阿里云 Key 也能点击录音并看到文本。
- 前端能显示中间结果和最终结果。

完成标准：

- 端到端演示不依赖外部云服务。

### 3.4 记录 Mock 模式指标

操作：

- Mock ASR 也记录首字延迟。
- Mock ASR 也记录最终延迟。
- Mock ASR session 也写入数据库。

验证方法：

- 完成一次 Mock 录音后，指标面板显示延迟数据。
- 数据库 session 记录包含延迟字段。

完成标准：

- 指标链路在接真实 ASR 前已经可用。

### 4.1 实现基础工作台 UI

操作：

- 实现主文本区。
- 实现录音按钮。
- 实现确认、复制、清空等基础操作。
- 实现指标面板入口。
- 实现热词管理入口。

验证方法：

- 页面第一屏就是工作台。
- 不需要进入二级页面才能开始录音。
- 浏览器控制台无运行时错误。

完成标准：

- UI 能支撑完整 MVP 流程。

### 4.2 修正视觉原型中的不稳定点

操作：

- 去掉或补齐原型中未定义变量对应的交互。
- 对未进入 MVP 的后处理模式不要做半成品入口。
- 保留场景切换，但只作为前端状态，不影响 ASR 核心链路。

验证方法：

- 点击所有可见按钮。
- 控制台不出现变量未定义错误。

完成标准：

- 原型被工程化后稳定可用。

### 4.3 实现麦克风授权

操作：

- 用户点击录音时请求麦克风权限。
- 权限通过后进入录音状态。
- 权限拒绝时展示明确错误。

验证方法：

- 首次点击录音，浏览器弹出权限请求。
- 拒绝权限后页面不崩溃。
- 再次点击可以重新尝试。

完成标准：

- 权限流程清晰。

### 4.4 实现音频采集

操作：

- 优先使用 AudioWorklet。
- 输出 16kHz 单声道 PCM。
- 每 50 到 100ms 发送一帧音频给后端。
- 如果 AudioWorklet 第一阶段阻塞，可临时用可行方案打通链路，并记录技术债。

验证方法：

- 后端日志能看到连续音频帧。
- 停止录音后不再发送音频帧。
- 长按或连续录音不会造成明显内存增长。

完成标准：

- 浏览器到后端的音频流稳定。

### 4.5 实现前端 WebSocket 状态机

操作：

- 明确 idle、connecting、recording、stopping、completed、error 状态。
- 录音按钮根据状态变化样式和可点击性。
- 出错后允许重试。

验证方法：

- 正常录音流程状态顺序正确。
- 后端断开时进入 error 状态。
- 用户可以再次开始录音。

完成标准：

- 用户不会卡在不可恢复状态。

## 11. 阶段 5：修正和本地热词闭环

### 5.1 维护原始识别文本和修正文本

操作：

- 前端保留 ASR 原始文本。
- 用户编辑的是修正文本。
- 提交修正时同时发送原始文本和修正文本。

验证方法：

- 用户把“张力”改成“张立”后，提交请求中同时包含两份文本。
- 没有修改时不允许提交修正。

完成标准：

- 后端有足够信息提取热词。

### 5.2 实现修正提交 API

操作：

- 后端接收 `user_id`、`session_id`、`original_text`、`corrected_text`。
- 计算编辑距离。
- 保存 correction 记录。

验证方法：

- 提交修正后数据库出现 correction。
- correction 包含原文、修正文、编辑距离和创建时间。

完成标准：

- 修正历史可追踪。

### 5.3 实现热词提取

操作：

- 从修正文本中提取候选正确词。
- 过滤单字。
- 过滤纯标点。
- 过滤过长短语。
- 中文热词长度控制在 2 到 10 个汉字。
- 对英文、数字和混合词保留合理长度限制。

验证方法：

- “张力 -> 张立”提取“张立”。
- “熔断基质 -> 熔断机制”提取“熔断机制”。
- 空修改不提取。
- 标点和单字不提取。

完成标准：

- 热词提取可解释、可测试。

### 5.4 合并和更新本地热词

操作：

- 同一用户同一热词只保留一条。
- 重复修正时增加 `correction_count`。
- 默认权重为 3。
- 多次修正后权重最高提升到 5。
- 更新 `updated_at`。

验证方法：

- 连续提交同一热词不会重复插入。
- 修正次数递增。
- 权重不会超过 5。

完成标准：

- 本地热词库能随用户修正持续成长。

### 5.5 标记热词表同步状态

操作：

- 用户新增、删除、修正热词后，把 `user_vocabularies.sync_status` 标记为 `dirty`。
- 如果用户还没有 `user_vocabularies` 记录，则创建。

验证方法：

- 提交修正后同步状态变为 `dirty`。
- 前端显示“待同步”或等价状态。

完成标准：

- 后台同步任务有明确触发依据。

## 12. 阶段 6：真实阿里云 ASR 接入

### 6.1 配置真实 ASR 模式

操作：

- 后端根据环境变量决定使用 Mock ASR 还是真实阿里云 ASR。
- 本地开发默认可以使用 Mock。
- 真实模式必须要求 `DASHSCOPE_API_KEY` 存在。

验证方法：

完成标准：

### 6.2 建立阿里云 WebSocket 连接

## 13. 阶段 7：热词表同步

### 7.1 实现同步 worker

操作：

- 后端启动一个单进程后台 worker。
- worker 定期查找 `dirty` 的用户热词表。

验证方法：

### 7.2 选择同步热词

完成标准：

### 7.3 创建或更新远端热词表

完成标准：

### 7.4 处理同步失败

操作：

## 19. 风险和处理策略

| 风险 | 影响 | 处理策略 |
| --- | --- | --- |
| 阿里云 ASR 接入卡住 | 影响真实识别演示 | 保留 Mock ASR，先保证产品闭环可演示 |
| 浏览器音频格式不符合要求 | 真实识别失败或延迟高 | 优先统一为 16kHz 单声道 PCM，必要时先用临时采集方案打通 |
| 首字延迟达不到 300ms | KPI 不达预期 | 如实记录指标，调整音频帧大小和静音参数，不伪造数据 |
| 热词同步失败 | 个性化能力延迟生效 | 本地热词继续保存，前端提示稍后同步 |
| SQLite 文件丢失 | 用户热词和指标丢失 | 部署时强制持久化 `data` 目录 |
| 前端状态复杂导致卡死 | 用户无法继续录音 | 明确状态机，所有错误状态都允许重试 |
| API Key 泄露 | 安全风险 | Key 只存在后端环境变量，日志脱敏，前端不持有 |

## 20. 推荐实施顺序

严格按以下顺序推进：

1. 建目录和环境变量约定。
2. 初始化前端和后端。
3. 打通 `/healthz`。
4. 建 SQLite 和数据模型。
5. 实现用户和热词基础 API。
6. 实现 Mock ASR WebSocket。
7. 实现前端录音工作台。
8. 打通 Mock 端到端流程。
9. 实现修正提交和热词提取。
10. 实现指标记录和展示。
11. 接真实阿里云 ASR。
12. 接热词表同步。
13. 增加测试。
14. 做 Docker Compose 部署。
15. 按最终验收清单逐项检查。

不要跳过 Mock ASR 直接接真实云服务。不要在 MVP 阶段引入账号系统、PostgreSQL、Redis、消息队列或复杂管理后台。
