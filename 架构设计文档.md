# VoxMem - 架构设计文档

## 1. 项目概述

### 1.1 项目简介

VoxMem 是一个基于实时语音识别、LLM 文本整理和本地纠错记忆的语音输入工作台。用户可以在浏览器中直接录音，系统实时返回转写结果，并在停止录音后生成可编辑文本。用户对文本进行修正后，系统会把修正记录保存到 SQLite，形成个人本地替换记忆，让后续输入逐步减少重复错误。

项目定位是“轻量、可演示、可持续学习”的语音输入 MVP，重点解决专有名词识别错误、口语文本不可直接使用、多人声音混入和评审现场可稳定演示的问题。

### 1.2 技术栈

- 前端：React 18 + TypeScript + Vite + lucide-react
- 后端：Go 1.25 + net/http + Gorilla WebSocket
- 数据库：SQLite（modernc.org/sqlite）
- 实时 ASR：阿里云 DashScope Paraformer WebSocket
- 文件转写：DashScope 文件转写 + 说话人分离
- 文件暂存：七牛 Kodo
- 文本整理：OpenAI-compatible LLM API
- 测试：Go test + Playwright
- 本地开发系统：Windows + PowerShell

### 1.3 核心功能特点

- 实时语音输入：浏览器采集麦克风音频，实时发送到后端 WebSocket。
- ASR 后端代理：API Key 只保存在后端，前端不直接访问 DashScope。
- 本地纠错记忆：用户编辑文本后，系统保存 `from_text -> to_text` 替换规则。
- 文本整理模式：支持 raw、polish、markdown 三种输出。
- 质量观察：展示修正次数、平均修改率、热词命中、最近修正等数据。
- 声纹过滤流程：通过完整 WAV 文件转写和说话人分离实现可选过滤。
- Mock ASR：测试和演示时不依赖真实云服务。
- 音频调试：可保存 PCM 和 WAV 文件，方便排查麦克风采集问题。

## 2. 系统架构

### 2.1 总体架构

```text
[ Browser / React ]
    |
    | HTTP / WebSocket
    v
[ Go API Server ]
    |
    +-- SQLite 本地记忆库
    |
    +-- DashScope Paraformer 实时 ASR
    |
    +-- OpenAI-compatible LLM 文本整理
    |
    +-- 七牛 Kodo 音频暂存
    |
    +-- DashScope 文件转写与说话人分离
```

### 2.2 前端架构（`web/`）

```text
web/
├── src/
│   ├── App.tsx              # 主工作台：录音、识别、文本编辑、热词、指标、声纹过滤
│   ├── main.tsx             # React 入口
│   ├── styles.css           # 页面样式
│   └── vite-env.d.ts        # Vite 类型声明
├── tests/e2e/
│   └── realtime-asr.spec.ts # Playwright E2E 测试
├── package.json             # 前端依赖和脚本
├── vite.config.ts           # Vite 配置
└── playwright.config.ts     # E2E 配置，包含 Mock 后端启动
```

前端职责：

- 管理用户浏览器 `user_id`。
- 检查后端健康状态。
- 获取麦克风权限并采集音频。
- 将音频降采样为 16kHz 单声道 PCM。
- 通过 WebSocket 发送音频帧并接收转写事件。
- 展示实时文本、最终输入文本、处理结果和错误状态。
- 提交用户修正，刷新本地热词。
- 展示输入质量指标和最近修正。
- 在声纹过滤模式下录制 WAV 并调用文件转写接口。

### 2.3 后端架构（`server/`）

```text
server/
├── cmd/
│   ├── server/              # Web API 服务入口
│   └── asr-probe/           # DashScope ASR 探针工具
├── internal/
│   ├── asr/                 # Paraformer 实时 ASR、文件转写客户端
│   ├── audiodebug/          # PCM/WAV 调试录音保存
│   ├── localenv/            # .env 和 .env.local 加载
│   ├── memory/              # SQLite 存储、热词映射、偏好记忆、指标
│   └── textproc/            # raw/polish/markdown 文本处理
├── go.mod
└── go.sum
```

后端职责：

- 加载 `.env` 和 `.env.local`。
- 提供 HTTP API 和 WebSocket 服务。
- 校验 CORS 和 WebSocket Origin。
- 代理 DashScope 实时 ASR，不暴露云端 API Key。
- 保存用户、纠错记录、本地替换映射和偏好。
- 根据输出模式调用 LLM 或直接返回本地增强文本。
- 上传完整 WAV 到七牛 Kodo，并提交 DashScope 文件转写。
- 输出结构化日志和调试音频。

### 2.4 数据库架构（SQLite）

当前 SQLite 用于本地 MVP，重点支持个人纠错记忆和指标观察。

核心数据：

```text
users
├── 用户 ID
├── 创建时间
└── 最近访问时间

hotword_mappings
├── user_id
├── from_text
├── to_text
├── correction_count
├── hit_count
└── 更新时间

corrections
├── user_id
├── session_id
├── original_text
├── enhanced_text
├── corrected_text
├── edit_distance
└── created_at

preferences
├── user_id
├── key
├── value
└── updated_at
```

数据库设计原则：

- MVP 阶段使用 SQLite，降低部署门槛。
- 本地替换映射按用户隔离。
- 修正记录保留原始文本、增强文本和最终修正文本，便于后续分析。
- 映射命中时更新 hit_count，用于衡量记忆是否真正生效。

## 3. 系统功能架构

### 3.1 用户功能分层

#### 3.1.1 普通使用流程

- 打开页面并进入工作台。
- 选择输出模式。
- 点击开始录音。
- 查看实时识别文本。
- 停止录音后编辑最终文本。
- 提交确认，系统学习修正。

#### 3.1.2 进阶使用流程

- 查看热词管理，删除不需要的替换规则。
- 查看输入质量观察，了解修改率和热词命中情况。
- 启用声纹过滤，处理多人声音混入场景。
- 使用 Markdown 模式整理结构化内容。

#### 3.1.3 开发和测试流程

- 使用 Mock ASR 模式快速验证页面和本地记忆。
- 使用 `asr-probe` 检查 DashScope 鉴权和任务启动。
- 使用音频调试目录保存 WAV，确认浏览器采集是否正常。
- 使用 Playwright 跑 E2E。

### 3.2 核心模块详细设计

#### 3.2.1 实时 ASR 模块

```text
实时 ASR 流程:
├── 前端请求麦克风权限
├── AudioContext 采集音频
├── 降采样到 16kHz 单声道 PCM
├── WebSocket 连接 /ws/asr?user_id=...&mode=...
├── 后端创建 DashScope ASR 任务
├── 前端持续发送二进制音频帧
├── 后端转发音频到 DashScope
├── DashScope 返回 result-generated 事件
├── 后端解析中间结果和最终结果
└── 前端展示实时转写和最终文本
```

设计考虑：

- 前端不持有 DashScope API Key。
- WebSocket 双工传输适合实时音频流。
- 后端记录音频帧数量和字节数，便于排查无声或格式错误。
- 支持 Mock 模式，降低演示和测试风险。

#### 3.2.2 本地纠错记忆模块

```text
纠错记忆流程:
├── ASR 得到 original_text
├── 后端应用已有 hotword_mappings 得到 enhanced_text
├── 前端展示可编辑 input_text
├── 用户修改并确认
├── 后端保存 correction
├── 提取 from_text -> to_text 映射
├── 写入 hotword_mappings
└── 下次识别时优先本地替换
```

设计考虑：

- 本地替换发生在 LLM 之前，保证后处理基于更准确的专有词。
- 映射按用户隔离，避免不同用户之间互相污染。
- 替换规则可删除，避免错误记忆长期影响输出。
- 命中次数和修正次数用于展示自学习效果。

#### 3.2.3 LLM 文本整理模块

```text
输出模式:
├── raw
│   ├── 不调用 LLM
│   └── 只应用本地替换记忆
├── polish
│   ├── 调用 LLM
│   ├── 清理明确自我纠正
│   ├── 删除明显口头禅和重复
│   └── 保守提升可读性
└── markdown
    ├── 调用 LLM
    ├── 识别步骤、要点、列表
    └── 输出 Markdown 正文
```

设计考虑：

- LLM 不处理实时中间结果，只处理最终文本或用户确认文本。
- raw 模式作为低成本、低依赖兜底路径。
- 提示词强调保守处理，不添加用户没有说过的事实。
- LLM 失败时错误可见，避免用户误以为处理成功。

#### 3.2.4 声纹过滤模块

```text
声纹过滤流程:
├── 前端启用去除周围人声
├── 录音时缓存完整音频
├── 停止后编码为 WAV
├── POST /api/transcribe/file
├── 后端上传 WAV 到七牛 Kodo
├── 后端提交 DashScope 文件转写
├── 获取 sentences 和 speaker_id
├── 前端按首个说话人保留文本
└── 用户可恢复显示全部文本
```

设计考虑：

- 该流程不是实时声纹识别，而是文件转写后的说话人分离。
- 需要七牛 Kodo 提供 DashScope 可访问的音频 URL。
- 作为可选能力，不阻塞基础实时输入。

#### 3.2.5 指标观察模块

```text
指标内容:
├── correction_count          # 累计修正次数
├── average_edit_rate         # 平均修改率
├── hotword_count             # 本地热词数量
├── hotword_hit_count         # 热词命中次数
├── hotword_learned_count     # 热词学习次数
└── recent_corrections        # 最近修正记录
```

设计考虑：

- 让用户和评委看到系统确实在学习。
- 修改率能反映输入结果距离可用文本的差距。
- 热词命中次数能说明本地记忆是否产生实际价值。

## 4. 接口设计

### 4.1 后端 HTTP / WebSocket 接口

| 方法 | 路径 | 说明 |
| --- | --- | --- |
| GET | `/healthz` | 健康检查 |
| GET | `/ws/asr` | 实时 ASR WebSocket |
| POST | `/api/input/commit` | 提交最终输入文本，执行学习和文本处理 |
| POST | `/api/correction` | 保存用户纠错记录 |
| GET | `/api/hotwords` | 查询当前用户本地替换映射 |
| DELETE | `/api/hotwords/{id}` | 删除本地替换映射 |
| GET | `/api/metrics/summary` | 查询输入质量汇总 |
| GET | `/api/preferences` | 查询格式偏好 |
| DELETE | `/api/preferences/{key}` | 删除格式偏好 |
| POST | `/api/transcribe/file` | 上传完整 WAV 并执行文件转写说话人分离 |

### 4.2 WebSocket 客户端事件

前端发送：

```json
{
  "type": "stop"
}
```

后端返回：

```json
{
  "type": "transcript",
  "text": "今天找张力确认方案",
  "final": false
}
```

```json
{
  "type": "processed",
  "text": "今天找张立确认方案。",
  "mode": "polish",
  "source": "llm",
  "latency_ms": 1200
}
```

### 4.3 文本提交接口

`POST /api/input/commit`

```json
{
  "user_id": "browser-generated-uuid",
  "session_id": "asr-session-id",
  "mode": "raw",
  "original_text": "今天找张力确认方案",
  "enhanced_text": "今天找张力确认方案",
  "final_text": "今天找张立确认方案",
  "learn_hotwords": true,
  "manual_edit_base": "今天找张力确认方案"
}
```

返回：

```json
{
  "status": "ok",
  "text": "今天找张立确认方案",
  "mode": "raw",
  "source": "local",
  "latency_ms": 1,
  "mappings": [
    {
      "from_text": "张力",
      "to_text": "张立"
    }
  ]
}
```

## 8. 风险与应对

| 风险 | 影响 | 应对 |
| --- | --- | --- |
| 云端 ASR 延迟不稳定 | 实时体验变差 | 保留实时状态和日志，使用 Mock 模式保证演示可控 |
| 麦克风采集无声或格式异常 | ASR 无结果 | 保存 PCM/WAV 调试文件，优先确认音频质量 |
| LLM 输出过度改写 | 用户不信任结果 | 提示词要求保守，保留原始识别和可编辑输入框 |
| 本地替换误伤 | 文本被错误替换 | 只从明确修正中学习，提供热词删除功能 |
| API Key 泄露 | 安全风险 | 前端不保存第三方 Key，统一由后端代理 |
| 声纹过滤依赖外部存储 | 配置复杂 | 作为可选能力，不影响主链路演示 |
| SQLite 并发能力有限 | 多用户扩展受限 | MVP 阶段接受，后续可替换为 PostgreSQL |
